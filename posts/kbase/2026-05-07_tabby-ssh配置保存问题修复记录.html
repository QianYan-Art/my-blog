Tabby SSH 配置保存问题修复记录 - 2026-05-07

结论

本机 Tabby 已修复“新建 SSH 配置点击保存无反应 / 保存后不出现在列表里”的问题。

当前本机状态已恢复到接近排障前，仅保留必要修复补丁；用户原有壁纸、插件、自定义配置已恢复。

问题现象

当时本机 Tabby 版本为 1.0.232。

主要现象：

  1. 新建 SSH 配置时，点击蓝色“保存”按钮无反应。
  2. 部分情况下弹窗能关闭，但新建配置不会出现在配置列表里。
  3. 直接手改 config.yaml 可生效，说明底层配置机制未完全损坏，问题主要在前端配置编辑链路。

已定位的根因

本机实际排查到 3 个问题：

  1. SSH 配置页对旧配置里的 algorithms: {} 不兼容。

    2. - 会触发： - this.profile.options.algorithms[k] is not iterable

  1. 配置编辑弹窗保存时无条件调用 profile.__cleanup()。

    2. - 新建配置对象不一定带这个方法。 - 会触发： - this.profile.__cleanup is not a function

  1. 配置编辑弹窗关闭时返回了旧对象，而不是当前编辑后的序列化结果。

    2. - 结果是弹窗关闭了，但新建 profile 没有真正落盘到配置列表。

本机保留的修复

本机当前仅保留 2 处安装目录补丁：

  1. D:\jetbrains\tabby\resources\builtin-plugins\tabby-ssh\dist\index.js
  2. D:\jetbrains\tabby\resources\builtin-plugins\tabby-settings\dist\index.js

修复作用：

  • 兼容旧 SSH 配置里的空 algorithms 数据。
  • 允许缺失 __cleanup 的 profile 保存。
  • 确保“新建 SSH 配置”保存时把实际编辑后的内容返回给上层，从而真正写入配置。

后续注意事项

1. 升级/重装 Tabby 后要重点检查

因为本机修复是直接改安装目录下的编译产物，所以以下行为可能覆盖补丁：

  • Tabby 自动更新
  • 手动重装 Tabby
  • 替换 D:\jetbrains\tabby 目录

如果未来升级后问题再次出现，优先检查这两个文件是否已被覆盖：

  • D:\jetbrains\tabby\resources\builtin-plugins\tabby-ssh\dist\index.js
  • D:\jetbrains\tabby\resources\builtin-plugins\tabby-settings\dist\index.js

2. 判断问题是否复发的最小验证方式

做一次最小测试即可：

  1. 打开 Tabby
  2. 新建一个临时 SSH 配置
  3. 点击保存
  4. 看是否立即出现在配置列表里

若不能出现，则说明保存链路再次损坏。

3. 不建议随手清空整个 Roaming\tabby

之前排障证明，问题不只是缓存，也涉及安装目录下插件产物逻辑。

因此未来若再次异常：

  • 优先先看是否为版本覆盖导致的补丁失效
  • 不要直接删除整份用户目录，避免丢失自定义设置、壁纸、插件、连接配置

后续补充 - netcup 条目异常

在主问题修复完成后，用户重新通过 Tabby UI 新建了 netcup 条目。

该条目一度出现“能保存、能出现在列表里，但连接时报错”的情况。实际落盘配置表现为：

  • auth 未明确写入
  • algorithms 被保存成全空数组

对应现象：

  • Agent auth selected, but no running Agent process is found
  • NoCommonAlgo ... ours: []

这说明：

  • 该问题不属于前面已提交 PR 的 3 个核心保存链路崩溃问题
  • 更像是新建 SSH profile 后，个别字段在某些 UI 路径下被保存成了不完整值

本机处理方式：

  • 直接修正 C:\Users\lenovo\AppData\Roaming\tabby\config.yaml 中该 profile：

    • - 删除空的 algorithms 覆盖 - 显式设置 auth: password

  • 重启 Tabby 重新读取配置

结论：

  • 这不会影响已经提交的上游 PR 结论
  • 若后续稳定复现“新建 SSH profile 后 auth/algorithms 落盘不完整”，应作为单独问题继续跟进，而不是并入当前 PR
