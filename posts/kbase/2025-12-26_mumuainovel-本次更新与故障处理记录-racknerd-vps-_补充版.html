1. 当前部署概览（关键事实）

1.1 主目录

  • 项目目录：/root/MuMuAINovel
  • 备份目录：/root/backup_mumu
  • 运维脚本目录：/root/mumu_ops

1.2 运行中的容器（关键容器名固定）

  • 应用：mumuainovel（镜像：mumujie/mumuainovel:latest）
  • 数据库：mumuainovel-postgres（镜像：postgres:18-alpine）

1.3 端口暴露策略

  • 应用：127.0.0.1:8000->8000/tcp（应用仅本机回环暴露，通常由反代/边缘服务对外）
  • Postgres：已调整为 仅回环暴露（目标效果：127.0.0.1:5432->5432/tcp）

1.4 应用健康检查

  • 轻量健康检查：curl -fsS http://127.0.0.1:8000/health

2. 今晚发生了什么（问题描述）

2.1 报错现象

应用日志出现数据库列不存在错误：

  • column characters.main_career_id does not exist
  • 触发位置：访问项目角色/人物相关接口时（如 GET /api/... 读取 characters 表）

2.2 直接原因

  • 镜像/代码更新后，后端 ORM/SQL 已开始读取 characters.main_career_id
  • 但 现有数据库结构未同步更新（缺少该列）

2.3 额外观察

在数据库容器内检查初始化脚本目录：

  • /docker-entrypoint-initdb.d 下只有 init.sql
  • 手动执行 /docker-entrypoint-initdb.d/init_postgres.sql 报错：文件不存在

> 备注：Postgres 官方镜像的 docker-entrypoint-initdb.d 脚本通常只在“首次初始化数据卷”时自动执行；数据库已经有数据卷时，新增脚本不会自动再跑。

3. 今晚实际做过的操作（按时间线）

3.1 确认容器状态

docker compose ps

看到：

  • mumuainovel 健康
  • mumuainovel-postgres 健康

3.2 尝试执行数据库脚本（失败）

docker exec -it mumuainovel-postgres sh -lc 'ls -lah /docker-entrypoint-initdb.d'

docker exec -i mumuainovel-postgres sh -lc 'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -f /docker-entrypoint-initdb.d/init_postgres.sql'

结果：init_postgres.sql 不存在。

3.3 发现实际报错

docker logs -n 200 mumuainovel

定位到：characters.main_career_id 列缺失。

3.4 停止应用，避免修表时并发读写

docker compose stop mumuainovel

3.5 备份数据库（强烈建议每次改表前都做）

mkdir -p /root/backup_mumu

docker exec -t mumuainovel-postgres sh -lc 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB"' \
> /root/backup_mumu/pg_$(date +%F_%H%M).sql

3.6 手动补齐缺失字段（修复核心）

cat <<'SQL' | docker exec -i mumuainovel-postgres sh -lc 'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB"'
ALTER TABLE characters
  ADD COLUMN IF NOT EXISTS main_career_id VARCHAR;
SQL

3.7 重启应用

docker compose up -d mumuainovel

之后再次观察日志，应用能够正常启动并出现：

  • “初始化数据库表结构…成功”

4. Postgres 端口改为本地回环（安全加固）

目标：避免 Postgres 直接对公网暴露。

4.1 修改方式（docker-compose.yml）

在 postgres service 的 ports: 配置中：

  • 从："5432:5432"
  • 改为："127.0.0.1:5432:5432"

4.2 应用更改

cd /root/MuMuAINovel
docker compose up -d --force-recreate postgres

验证：

docker compose ps

应该看到 Postgres 的端口绑定只在 127.0.0.1。

5. Watchtower 自动更新现状与本次调整

5.1 Watchtower 当前配置（从 inspect 得到）

  • 容器：watchtower
  • 计划：--schedule "0 0 4 *"（每天 04:00 触发，具体以 watchtower 容器内 TZ 为准）
  • 参数：--cleanup（更新后清旧镜像）
  • restart：always
  • 挂载：/var/run/docker.sock:/var/run/docker.sock

5.2 本次改动：给 Postgres 打上“跳过更新”标签

在 docker-compose.yml 的 postgres service 内增加：

labels:
  - "com.centurylinklabs.watchtower.enable=false"

应用：

cd /root/MuMuAINovel
docker compose up -d --force-recreate postgres

验证：

docker inspect mumuainovel-postgres --format '{{json .Config.Labels}}' | grep -i watchtower

> 备注：不同 watchtower 运行模式对 label 规则生效方式略有差异。为了确认“数据库不会被更新”，最可靠的做法是：观察 watchtower 日志是否跳过该容器。

检查 watchtower 最近日志：

docker logs -n 200 watchtower

（可选）临时手动触发一次检查：

docker exec watchtower /watchtower --debug --run-once

11. v1.2.0 大版本升级（Alembic 迁移体系）与“重建数据库”处理

11.1 背景与决策

作者发布新版本（v1.2.0）后，数据库初始化/迁移机制发生较大变化（引入 Alembic）。对于“首次升级到该大版本”，作者建议按文档进行 重新初始化数据库。

本次决策：不保留历史数据，采用“清库重建”的升级路径（更干净、后续迁移更省心）。

> ⚠️ 注意：清库重建会删除 Postgres 的数据卷（命名卷 postgres_data），不可逆。执行前建议仍保留一份备份文件以便救援/回溯。

11.2 升级步骤（清库重建）

建议在项目目录执行：

1）临时停用 watchtower，避免更新过程中被自动更新打断：

docker stop watchtower

2）备份数据库（即使不打算保留数据，也建议留档）：

mkdir -p /root/backup_mumu

docker exec -t mumuainovel-postgres sh -lc 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB"' > /root/backup_mumu/pg_$(date +%F_%H%M).sql

3）拉取最新镜像：

cd /root/MuMuAINovel
docker compose pull

4）清库重建（删除数据卷）：

docker compose down -v

5）重新启动服务：

docker compose up -d

6）验证服务可用（健康检查 + 日志）：

docker compose ps
curl -fsS http://127.0.0.1:8000/health && echo OK
docker logs -n 200 mumuainovel

7）恢复 watchtower：

docker start watchtower

11.3 升级后状态确认要点

  • docker compose ps 显示应用与数据库为 Up/healthy
  • /health 返回 200
  • 应用日志不再出现 UndefinedColumn / does not exist / relation ... does not exist 等数据库结构类错误

12. mumu_ops 运维脚本：现状、用途与“什么时候用哪个”

12.1 当前脚本清单（建议固定路径）

  • 体检：/root/mumu_ops/mumu-check.sh
  • 备份：/root/mumu_ops/mumu-db-backup.sh
  • 恢复：/root/mumu_ops/mumu-db-restore.sh
  • 日常全量更新（不清库）：/root/mumu_ops/mumu-full-update.sh
  • 大版本重建库升级（清库）：/root/mumu_ops/mumu-resetdb-upgrade.sh

12.2 什么时候用哪个（最短记忆版）

1）登录先检查 / 怀疑出问题

/root/mumu_ops/mumu-check.sh
  • 作用：检查容器状态、健康检查、日志关键字，并在末尾打印“建议你直接复制执行的命令”。

2）日常更新（绝大多数情况）

/root/mumu_ops/mumu-full-update.sh
  • 适用：小更新、普通版本升级、watchtower 更新后你想手动再跑一次确保一致。
  • 特点：会先备份、拉更新、重启、health 检查，并在更新期间临时 stop watchtower，完成后再恢复，避免撞车。

3）作者明确说“需要重新初始化数据库 / down -v” 或你愿意清库

/root/mumu_ops/mumu-resetdb-upgrade.sh
  • 适用：类似 v1.2.0 这种“数据库体系大改”的版本。
  • 特点：包含锁与退出保护（脚本失败也会尽量把 watchtower 恢复），核心动作是 docker compose down -v（清库）。

4）回滚（数据库恢复）

/root/mumu_ops/mumu-db-restore.sh
  • 适用：恢复某个备份 dump。
  • ⚠️ 注意：如果恢复的是“旧版本数据库”，通常需要同时回滚应用版本，否则可能出现新代码不兼容旧库。

14. 本次“手动补字段”与“清库重建”的关系说明（避免以后混淆）

  • “手动补字段（ALTER TABLE … ADD COLUMN）”是对 某次更新引发的单字段不兼容 的快速修复手段。
  • “清库重建（down -v）”是对 数据库体系大改（如 v1.2.0 引入 Alembic） 的一次性升级手段。
  • 后续如果版本都在 Alembic 体系内，理论上应该更多依赖应用自带迁移，而不是继续手动补字段。
