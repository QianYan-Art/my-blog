Git本机环境说明与后续操作准则

2026-03-176 min

文档用途

这份文档用于给后续接手的大模型或人工维护者快速建立上下文，避免在不了解本机 Git 历史和现状的前提下误改环境。

适用场景：

  • 终端或 IDE 出现 Git 版本、路径、LFS、凭据相关问题
  • 需要升级 Git
  • 需要判断是否可以删除旧 Git
  • 需要排查某个项目为何仍然命中旧 Git 路径

关键信息结论

截至 2026-03-16，本机 Git 已完成从旧版手动安装 Git 向 Scoop 管理版 Git 的主路径切换。

  • 默认应使用的 Git：C:\Users\lenovo\scoop\shims\git.exe
  • 默认目标版本：2.53.0.windows.2
  • 旧 Git 路径：D:\jetbrains\git\Git\cmd\git.exe
  • 旧 Git 来源：不是 JetBrains 自带，是用户早年手动下载官网安装包后放到 D:\jetbrains\git，期间可能用过 choco 做过一次更新

核心原则：

  • Git 仓库不绑定某个 git.exe 路径
  • 切换 Git 可执行文件通常不会破坏已有仓库
  • 真正需要关注的是终端 PATH、IDE 配置、脚本里是否写死旧路径

当前推荐的事实判断方式

后续排查时，不要只看单一命令结果，要分层判断。

第一层：看当前终端实际命中哪个 Git

where.exe git
git --version

第二层：看注册表中的真实 PATH

reg.exe query "HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment" /v Path
reg.exe query HKCU\Environment /v Path

第三层：看 PowerShell 启动脚本是否覆盖了 PATH 顺序

检查文件：

  • C:\Users\lenovo\Documents\PowerShell\Microsoft.PowerShell_profile.ps1

第四层：看 Git 配置来源

git config --show-origin --list
git config --show-origin --get credential.helper
git lfs version

对后续模型的操作准则

如果未来要继续处理本机 Git，请按下面顺序做，不要跳步。

准则 1：先确认问题属于哪一层

优先区分下面几类问题：

  • 只是终端 PATH 顺序问题
  • 某个 IDE 单独配置了旧 Git 路径
  • 某个脚本写死了旧路径
  • Git LFS / 凭据 / SSH / GPG 的兼容问题
  • 真实的仓库损坏问题

不要一上来就删除 Git 或重装 Git。

准则 2：不要把“仓库问题”和“Git 安装路径问题”混为一谈

项目仓库通常不会因为 git.exe 路径切换而损坏。

真正会受影响的是：

  • JetBrains 系列 IDE 中的 Git 可执行文件路径
  • VS Code / 其他编辑器的手动 Git 路径配置
  • bat / ps1 / cmd / 自定义工具中的硬编码路径

准则 3：默认优先维护 Scoop 这套 Git

后续如果需要升级 Git，默认应优先使用：

scoop update git

不要优先回到 D:\jetbrains\git\Git\cmd\git.exe 那套手动 Git。

准则 4：删除旧 Git 前先完成三项验证

如果以后要彻底删除旧 Git 目录，先确认：

  1. 新终端中 git --version 稳定返回 Scoop 新版
  2. 常用仓库内的 git status、git pull、git push、git lfs 正常
  3. JetBrains / VS Code 等常用 IDE 提交、拉取、比较、历史查看正常

如果三项没做完，不要直接删除旧目录。

准则 5：如果发现结果互相矛盾，优先相信“新开终端 + 注册表 + 配置文件”

某些长驻进程、旧终端、集成终端可能持有过期环境变量。

如果出现下面这种情况：

  • reg query 看起来已经移除了旧路径
  • 但某个当前会话里 where.exe git 仍能看到旧路径

先不要急着回滚，优先：

  1. 关闭并重新打开终端
  2. 重启 IDE
  3. 再重新执行 where.exe git 和 git --version

IDE 与脚本排查建议

如果未来仍有项目或工具命中旧 Git，优先排查这些位置：

  • JetBrains 设置中的 Path to Git executable
  • VS Code 用户设置中的 git.path
  • 项目内 .bat / .cmd / .ps1 / CI 脚本
  • 用户自己的工具启动脚本或快捷方式

JetBrains 推荐配置：

  • 直接填 C:\Users\lenovo\scoop\shims\git.exe
  • 或者填 git，前提是 IDE 启动时继承的是正确 PATH

推荐的最小排障命令集

where.exe git
git --version
git lfs version
git config --show-origin --list
git config --show-origin --get credential.helper
reg.exe query "HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment" /v Path
reg.exe query HKCU\Environment /v Path

如果是项目内排查，再补：

git status
git remote -v
git branch -vv

不建议的危险操作

后续若无充分证据，不建议直接做下面这些事：

  • 直接删除 D:\jetbrains\git\Git
  • 直接覆盖 .git 目录
  • 盲目执行 git gc --prune=now
  • 在不了解来源的情况下改写 .gitconfig
  • 未做备份就修改系统级 PATH

本次处理后的目标状态

理想状态应长期保持为：

  • 终端默认 Git 指向 C:\Users\lenovo\scoop\shims\git.exe
  • Git 升级通过 Scoop 完成
  • 旧 Git 不再参与 PATH 优先级
  • 各 IDE 如需显式配置 Git 路径，统一指向 Scoop Git

给后续模型的简短指令

如果你是后续接手的大模型，请先接受以下前提：

  • 不要把仓库问题误判为 Git 安装路径问题
  • 不要默认删除旧 Git
  • 先验证 PATH、注册表、PowerShell profile、IDE 配置
  • 默认维护 Scoop 这套 Git
  • 做系统级修改前先备份
