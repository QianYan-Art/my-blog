2. 统一更新命令

两台机器的日常操作保持一致。

/root/update_x-ui.sh status
/root/update_x-ui.sh prepare-cert
/root/update_x-ui.sh update

常用含义：

  • status：查看 3x-ui 服务、证书和面板访问状态。
  • prepare-cert：只检查或重建自签证书，不更新 3x-ui。
  • update：先备份，再检查证书，然后调用官方更新脚本，最后等待服务和本地 HTTPS 恢复。

从本机直接发起更新：

ssh -i "C:\Users\lenovo\.ssh\60.157.94.240_ssh.key" -p 27721 root@nat-jp-3h8e.aproxy.top "/root/update_x-ui.sh update"
ssh -i "C:\Users\lenovo\.ssh\141.239.74.52_ssh.key" -p 35560 root@nat.qq.pw "/root/update_x-ui.sh update"

3. 3x-ui 更新脚本设计

两台机器都使用同样的脚本结构：

/root/update_x-ui.sh        # 主入口，设置 PANEL_HOSTNAME
/root/update_x-ui.core.sh   # 通用逻辑，负责备份、证书、更新、健康检查

入口脚本只做主机名绑定。

日本 NAT 节点：

#!/bin/bash
export PANEL_HOSTNAME="nat-jp-3h8e.aproxy.top"
exec /root/update_x-ui.core.sh "$@"

qq.pw NAT 节点：

#!/bin/bash
export PANEL_HOSTNAME="nat.qq.pw"
exec /root/update_x-ui.core.sh "$@"

核心脚本的行为：

  • 失败即退出，避免安装失败后继续打印“成功”。
  • 更新前备份 /etc/x-ui、/usr/local/x-ui 和 x-ui.service。
  • 备份目录为 /root/x-ui-update-backups/。
  • 自签证书默认放在 /root/cert/${PANEL_HOSTNAME}/。
  • 证书不存在、已经过期或 30 天内到期时，自动重建并重新绑定到 3x-ui。
  • 更新时调用官方 update.sh，不调用 install.sh 冒充更新。
  • 不申请 Let's Encrypt 证书，不碰 80 端口。
  • 不修改面板用户名、密码、端口和数据库字段。
  • 官方更新后如果出现短暂 SQLite 锁，会等待 systemd 自动拉起并轮询本地 HTTPS。

这套脚本是为 NAT 机器准备的。核心原因是 NAT VPS 通常不能稳定控制公网 80 端口，因此不能把 3x-ui 更新流程绑定到 Let's Encrypt HTTP 验证上。

4. 证书策略

当前不是 Let's Encrypt 证书，而是面板专用自签证书。

证书文件：

/root/cert/<外部域名>/fullchain.pem
/root/cert/<外部域名>/privkey.pem

自签证书不会像 Let's Encrypt 那样自动续签。现在采用的是“月度自检”：

  • 每月自动执行一次 /root/update_x-ui.sh prepare-cert。
  • 如果证书还有超过 30 天有效期，就不做任何改动。
  • 如果证书 30 天内到期或已经失效，就自动生成新证书并重启 3x-ui。

手动检查：

/root/update_x-ui.sh prepare-cert
openssl x509 -in /root/cert/<外部域名>/fullchain.pem -noout -dates

6. qq.pw 小内存机器状态

nat.qq.pw 只有 256 MiB 级别内存，已经做过 swap，并按当前需要保留核心服务。

保留：

  • x-ui
  • xray
  • komari-agent
  • ssh
  • qemu-guest-agent
  • serial-getty@ttyS0

已停用：

  • exim4
  • getty@tty1

保留 qemu-guest-agent 和 serial-getty@ttyS0 是为了宿主面板兼容和救援通道。它们占用不大，优先级低于可恢复性。

后续不要随手关闭 komari-agent 和 x-ui。这两个服务是当前机器的核心用途。

7. 历史问题与保留结论

7.1 3x-ui 更新卡在 SSL 流程

早期在 nat.qq.pw 上尝试安装或更新 3x-ui 时，遇到过 SSL 证书流程卡住的问题。当时原因是：

  • NAT VPS 没有独立公网 80 端口。
  • 平台域名不可控，不适合做 Let's Encrypt HTTP 验证。
  • 官方安装或更新流程在没有证书时可能进入证书申请逻辑。

当时曾临时退回过 3x-ui v2.8.5，这个结论现在已经过时。

当前结论：

  • 两台机器都已使用 3x-ui v2.8.11。
  • 不再通过旧版安装脚本绕开 SSL。
  • 不再使用调用 install.sh 的自定义更新脚本。
  • 统一用自签证书 + 官方 update.sh 的方式更新。

7.2 APT 依赖异常

nat.qq.pw 曾在 kejilion 脚本更新后出现 APT 依赖异常：

  • cron 装不上。
  • linux-image-cloud-amd64 依赖不完整。
  • dpkg --configure -a 报 linux-image-6.1.0-44-cloud-amd64 缺失。

当时修复顺序：

dpkg --configure -a
apt update
apt --fix-broken install -y
apt install -y cron ufw
apt install -y curl wget tar socat

后续遇到类似问题，优先按这个顺序收敛，不要一上来强制卸包。

7.3 Shadowsocks 节点超时

曾出现过 Shadowsocks 入站创建后 Clash 一直超时的问题。排障顺序保留如下：

  1. 确认 3x-ui 里 Xray 是否正在运行。
  2. 确认 SS 入站监听端口是否存在。
  3. 确认 NAT 平台外部端口是否正确映射到内网监听端口。
  4. 确认 Clash 配置里填写的是公网映射端口，不是内网端口。
  5. 如果作为链式代理落地节点使用，再确认前置入口节点是否可用。

服务器检查命令：

ss -lntup | grep -E 'x-ui|xray|35561|31927'
journalctl -u x-ui -n 100 --no-pager

8. Clash 链式代理记录

原有 Clash / Clash Verge 设计仍有记录价值：

  • 本地 Clash 先走机场前置节点。
  • 前置节点连接 NAT VPS 的外部映射端口。
  • NAT VPS 作为日本家宽落地出口访问目标站点。

核心原则：

  • Clash 中填写的端口必须是 NAT 平台外部端口。
  • 3x-ui 中监听的是内网端口。
  • NAT 平台负责外部端口到内网端口的映射。
  • 如果节点作为链式落地使用，不要只测直连，还要测 relay 链路。

9. 防火墙与安全

NAT VPS 的主要边界是平台端口映射，不是传统公网防火墙。

建议：

  • 不需要的公网映射端口直接在平台后台删除。
  • SSH 映射端口只在维护时开放，维护完再关闭或限制。
  • 面板端口使用 HTTPS，即使是自签证书，也比纯 HTTP 更适合长期管理。
  • 文档不要明文写面板用户名、密码、WebBasePath 和 SS 密码。
  • 日本 NAT 节点的 cliproxyapi 上游已经收口到 127.0.0.1:59998；以后不要再额外对公网映射 59998，只保留前置代理使用的既有外部入口。

10.1 每日自动清理备份

2026-04-24 已在两台 NAT VPS 上部署每日自动清理任务：

/usr/local/sbin/cleanup-root-backups.sh
/etc/systemd/system/cleanup-root-backups.service
/etc/systemd/system/cleanup-root-backups.timer

cleanup-root-backups.service 是 oneshot 服务，只负责执行一次清理；cleanup-root-backups.timer 才负责每日调度，已执行 systemctl enable --now cleanup-root-backups.timer。timer 配置了 Persistent=true，如果 VPS 在计划时间离线，恢复后会补跑一次。

  • 日本 NAT 节点：清理 /root/cliproxyapi_backups/、/root/x-ui-update-backups/，每个目录仅保留最新 3 份备份。
  • qq.pw NAT 节点：当前只部署 3x-ui，清理 /root/x-ui-update-backups/，仅保留最新 3 份备份。
  • 兼容历史旧目录：脚本也会扫描 /root/x-ui-backup-*，最多保留最新 3 个。
  • 脚本只允许删除上述白名单路径，避免误删 /root 下的其他目录。

手动验证：

systemctl start cleanup-root-backups.service
systemctl status cleanup-root-backups.service --no-pager
systemctl list-timers cleanup-root-backups.timer --no-pager
journalctl -u cleanup-root-backups.service -n 80 --no-pager

如果更新后面板打不开，最短排障流程：

systemctl status x-ui --no-pager
journalctl -u x-ui -n 120 --no-pager
/root/update_x-ui.sh status
/root/update_x-ui.sh prepare-cert

11. 文档结构建议

当前先维持“总览 + 单机专题”的结构：

  • 本文继续作为两台 NAT VPS 的共同维护总览，集中记录共享脚本、定时任务、备份清理、证书策略和通用排障结论。
  • JP日本 SoftBank 家宽 + Clash 链式代理搭建指南.md 继续作为日本 NAT 节点的单机专题，记录 JP 节点的链式代理、端口映射和本机定时任务。
  • 如果后续 qq.pw 节点也积累了 Clash、端口映射或专项排障内容，再单独拆出 qq.pw NAT 节点维护记录.md，避免本文继续膨胀。

不建议现在强行改成“一篇文档写一个 VPS”，因为 3x-ui 更新脚本、自签证书策略、备份清理任务在两台机器上高度一致，拆散后容易出现两边文档不同步。

12. 一句话原则

这两台 NAT 家宽机器后续维护时，不要再让 3x-ui 更新依赖 80 端口或 Let's Encrypt。统一使用自签证书、自检 cron 和 /root/update_x-ui.sh update。
