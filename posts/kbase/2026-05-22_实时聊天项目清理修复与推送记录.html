4.3 识别出真正的残留问题

本次最关键的发现不是业务代码，而是仓库卫生问题：

#### 问题 1：backend/target/ 仍被 git 追踪

后果：

  • 每次运行或编译后端，工作区都会被 class 文件、jar、maven-status 等产物污染。
  • 这类文件已经被 .gitignore 覆盖，但因为曾被纳入版本库，所以 ignore 不会自动生效。

#### 问题 2：backend/logs/ 仍被 git 追踪

后果：

  • 每次运行后端，日志文件都会改动，导致 git status 变脏。
  • 这类日志本应是纯运行产物，不应参与版本管理。

#### 问题 3：frontend/server.js 保留未使用的 /ws 代理

后果：

  • 不影响当前运行，但会误导后续维护者。
  • 容易让人误以为本地 WebSocket 还需要走 Express 代理。

11. 后续排 bug 建议步骤

如果后续再出现“项目似乎没问题，但运行又不对”的情况，建议按下面顺序排：

第一步：先看 git 状态

执行：

  • git status

确认是不是又把不该追踪的文件带回版本库，或者是否真的只是运行产物变化。

第二步：查端口占用

执行：

  • netstat -ano | findstr ":3000"
  • netstat -ano | findstr ":8083"

避免把旧进程占端口误判成“代码启动失败”。

第三步：最小编译/语法验证

执行：

  • cd backend && mvn compile -q
  • cd frontend && node -c js/app.js && node -c js/websocket.js && node -c js/webrtc.js && node -c server.js

先分清是编译/语法问题，还是运行期问题。

第四步：检查前端 WebSocket 配置

重点核对：

  • frontend/js/config.js
  • WebSocket 本地应仍是 ws://localhost:8083/ws

不要误改回通过 Express /ws 代理的老思路。

第五步：检查认证链路

重点核对：

  • HTTP 是否带 Authorization: Bearer <token>
  • WebSocket CONNECT 帧是否也带 token
  • 登录失败时是否清理旧 token
  • 登出后是否断开 WebSocket 并清除本地状态

第六步：再做浏览器级联调

如果编译都过了但功能仍异常，再进入浏览器联调：

  1. 看 Console 报错
  2. 看 Network 中 /api/auth/login 返回
  3. 看 WebSocket 是否成功 CONNECT
  4. 看 STOMP SUBSCRIBE / SEND 帧是否发出
  5. 看消息 DOM 是否真实更新，而不是只看页面表象
