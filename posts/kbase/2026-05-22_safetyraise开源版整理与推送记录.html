SafetyRAISE开源版整理与推送记录

2026-05-229 min

2026-05-22_SafetyRAISE开源版整理与推送记录

1. 目标

在不影响私有部署主线的前提下，将 D:\MCP_Server\TS_analysis_report 当前较新的部署版代码整理为可公开发布的开源版，并推送到开源仓库 QianYan-Art/SafetyRAISE-OS。 本次记录只保留“从部署版整理为开源版并推送”的过程细节，私有部署版的日常开发、服务器维护、现网切换等内容仍以项目内 项目文档 目录为准。

2. 项目与仓库信息

  • 本地项目目录：D:\MCP_Server\TS_analysis_report
  • 私有部署仓库：QianYan-Art/SafetyRAISE.git
  • 开源仓库：QianYan-Art/SafetyRAISE-OS.git
  • 私有部署主线：codex/server-deploy
  • 私有主分支：main
  • 开源主分支：main
  • 开源快照标签：opensource

本次开源整理的基线不是旧的 opensource 快照内容，而是直接以当前部署版较新的代码状态为基础，再做一次面向公开仓库的脱敏、裁剪、重写和验证。

3. 开源整理前的判断

开始前先确认了一个原则：不能把私有部署仓库里的现状直接原样推到开源仓库。主要原因有四类。

3.1 私有维护文档不适合公开

项目目录下原有 项目文档 里的内容大量面向本机与服务器维护，包含：

  • 服务器路径与宿主机挂载结构
  • 远端分支与推送约定
  • 线上 Nginx 实际生效位置
  • SSH、切换流程、维护习惯

这些内容对开源用户没有直接帮助，反而会泄露维护结构与内部习惯，因此不能进入开源仓库主线。

3.2 配置文件混有私有部署信息

整理前，配置中存在以下几类不适合公开的内容：

  • 真实 API Key
  • 私有域名和私有端点
  • 本机绝对路径
  • 私有部署默认值

如果只删文档、不改配置，开源仓库仍然会带来安全风险与误导。

3.3 README 与说明文档不符合开源仓库口径

原始文档更像“当前维护者的工作备忘录”，不适合作为开源仓库首页。 开源仓库需要明确回答：

  • 项目是什么
  • 提供哪些能力
  • clone 后怎么启动
  • 需要准备哪些模型和运行资产
  • 哪些能力依赖本地模型、哪些依赖外部 API

3.4 旧开源仓库已有历史内容，但不能直接沿用

SafetyRAISE-OS 仓库原来放的是更早的 opensource 快照。 经过检查后决定：旧内容不再作为新的开源基线，但可以保留“旧快照曾存在”的历史认知；新的开源版应直接以当前部署代码整理生成。

4. 开源版整理的实际步骤

4.1 先从当前部署代码抽出开源整理线

整理时采用的思路不是改坏私有部署线，而是先从部署代码抽出一条开源整理线，集中完成下面几类工作：

  1. 删除不适合公开的文件
  2. 重写面向公开仓库的文档
  3. 拆分并脱敏配置
  4. 补齐开源用户需要的最小样例
  5. 运行验证
  6. 推送到 origin-os/main

4.2 删除不适合进入开源仓库的内容

开源整理阶段移除了下列内容或同类内容：

  • 项目文档/ 整个目录
  • 私有运维相关更新脚本
  • start-all.bat
  • 不必要的测试文件与仅用于本地维护的文件
  • 面向私有部署维护的脚本或说明

这里的原则很明确： 开源仓库只保留开源用户真正需要的代码、配置模板、最小样例和公开说明，不保留开发期间的内部维护资料。

4.3 配置脱敏与默认值重设

这一部分是开源整理的核心。

处理动作包括：

  1. 去掉配置文件中的真实 OPENROUTER_API_KEY
  2. 去掉配置文件中的真实 QIANYAN_API_KEY
  3. 去掉配置文件中的真实 DUCKCODING_API_KEY
  4. 统一改成读取环境变量
  5. 将不适合公开的私有默认值改成通用或示例值

除了 API Key，还同步整理了：

  • 专家模型默认配置
  • 报告模型三档端点说明
  • YOLO、embedding、reranker 的默认说明
  • 本机绝对路径改为公开可理解的示例路径或仓库内相对约定

4.4 重写 README 与公开文档

开源版重新梳理了以下文档：

  • README.md
  • docs/quickstart.md
  • docs/configuration.md
  • docs/deployment.md
  • docs/prepare-runtime-assets.md

这几份文档的职责分别被重新定义为：

  • README.md：仓库首页，只讲项目定位、能力、快速入口
  • quickstart.md：安装依赖、启动顺序、最小联调
  • configuration.md：环境变量、模型端点、运行配置说明
  • deployment.md：通用部署说明，不写私有运维口径
  • prepare-runtime-assets.md：模型、权重、知识库和运行资产准备方式

后续又针对文风做过一次收紧，避免出现“给维护者看的备忘语气”，统一调整为“给开源用户看的说明文风”。

4.5 增加开源用户最关心的运行资产说明

这是后来重点补的一次。

为了避免其他人 clone 后不知道该下载什么、怎么补知识库，开源版专门补齐了以下内容：

  1. YOLO 模型需要什么
  2. embedding 模型需要什么
  3. reranker 模型需要什么
  4. 专家小模型如何接入
  5. 报告模型三档端点在哪里改
  6. 知识库的最小文件格式是什么
  7. dense 检索资产需要哪些文件

同时新增了最小知识库样例，帮助使用者理解文件结构，而不是只看抽象描述。

4.6 专家小模型公开链接接入

在专家小模型准备发布到 Hugging Face 后，开源仓库侧做了两件事：

  1. 默认模型名更新为：

    2. - suyuan37/SafetyRAISE-TS-Qwen3

  2. 文档中同步说明：

    3. - 这里填的是“模型名”和“推理服务加载后的模型标识” - 不是直接把 Hugging Face 网页地址塞进程序

随后又检查了程序默认值是否和文档一致，确认了：

  • 专家模型默认 provider
  • 专家模型默认 base_url
  • 专家模型默认 model

这三者在开源版里保持一致，不再出现“文档说一种方式，程序默认却是另一种协议”的问题。

4.7 报告模型三档端点重新核对

开源版整理过程中，专门复查了报告模型的三档配置：

  • max
  • pro
  • lite

确认点包括：

  1. 配置文件里三档端点都存在
  2. 程序侧选择逻辑正常
  3. 前端公开配置只暴露 max / pro / lite
  4. 不直接向前端暴露供应商名
  5. 文档中明确写清每档在哪改、如何换端点

这一步是为了确保开源用户不会因为文档不清或默认值错位而误配报告模型。

5. 开源验证过程

整理不是只改文档，期间实际做过多轮验证。

5.1 基础验证

  • 后端配置加载验证
  • FastAPI 应用导入验证
  • 前端 npm run build 验证
  • 本地健康检查接口验证
  • app-config 接口验证

5.2 运行资产相关验证

针对开源版额外验证了：

  • 最小知识库样例是否能被读取
  • 检索器是否能在样例数据上返回结果
  • /ready 中缺失运行资产时的诊断信息是否合理

确认结果是： 开源仓库可以正常启动和暴露基础接口，但完整报告生成仍然依赖用户自行准备知识库、YOLO 权重、本地模型服务和外部 API，这一点在文档中已经明确说明。

5.3 文档与程序一致性验证

后续又专门做过一次“不要只查文档，也查程序”的复查，重点确认：

  • 专家模型默认值是否和文档一致
  • 专家模型 provider 是否和默认接入方式一致
  • 报告模型三档端点是否和文档一致
  • .env.example 是否和配置文件说明一致

6. 推送到开源仓库后的安全问题与修复

这一步非常关键，必须单独记录。

6.1 问题发现

虽然开源版当前工作树已经是脱敏后的干净状态，但 GitHub 后续发来了告警： SafetyRAISE-OS 仓库历史中暴露了真实 OpenRouter Key。

排查后确认：

  • 不是“当前文件又传错了”
  • 而是“开源仓库之前的历史提交里仍然残留了真实 key”

也就是说，当前树干净，不代表历史干净。

6.2 根因

开源仓库在整理初期虽然已经改掉了当前配置文件中的真实 key，但仍然沿用了早先的提交历史。 那些旧提交里包含过真实的：

  • OpenRouter Key
  • QianYan Key
  • DuckCoding Key

因此 GitHub secret scanning 仍然会报警。

6.3 处理方式

这里没有继续做“修当前文件”这种表面动作，而是直接处理根因：

  1. 切回开源分支 main
  2. 以当前已经脱敏的工作树创建全新根提交
  3. 重写开源仓库 main 的公开历史
  4. 强推到 origin-os/main
  5. 重新更新开源快照标签 opensource

这样处理后，开源仓库当前树和公开历史都不再包含旧的真实 key。

6.4 修复后的确认

修复后重新检查了：

  • 当前开源树
  • 新的开源远端历史
  • 开源快照标签

确认新的开源仓库历史不再命中旧的 key 特征。 也就是说，本次开源整理最终不是“只洗工作树”，而是把公开历史也一起洗干净了。

7. Hugging Face 专家模型仓库的补充情况

开源版整理过程中，专家小模型同步发布到了 Hugging Face：

  • https://huggingface.co/suyuan37/SafetyRAISE-TS-Qwen3

上传权重后，Hugging Face 一度把两个 safetensors 分片判为 unsafe。 后续通过邮件提交误报复核请求，官方回复已修复，因此这一问题最后不构成开源发布阻塞。

这里记录它的目的不是作为开源仓库主问题，而是说明开源版文档里引用的专家模型仓库已经走完一次可用性核验，不是未完成占位链接。

8. 开源版最终推送结果

开源整理收束后，开源仓库最终状态为：

  • 开源主分支：origin-os/main
  • 开源快照标签：opensource

两者已对齐到同一份整理完成且安全清理后的提交。 因此后续如果需要回看“开源版稳定快照”，优先看 opensource；如果继续沿开源主线演进，则以 origin-os/main 为准。

9. 与私有部署线的关系

本次开源版整理完成后，私有部署线与开源线形成了明确分工：

9.1 私有部署线负责

  • 现网部署
  • 服务器运维
  • 私有文档
  • 内部调优
  • 私有端点和实际生产配置

9.2 开源线负责

  • 可公开展示的项目能力
  • 可复现的最小工程结构
  • 示例配置与说明文档
  • 模型、知识库、运行资产接入方式

这样做的好处是：

  1. 不再把公开展示和私有运维混在同一仓库语境里
  2. 部署线可以继续快速迭代
  3. 开源线可以保持更清晰、可解释、可复现

10. 后续开发与维护建议

10.1 开源线后续改动原则

以后如果继续更新开源仓库，建议始终遵守以下原则：

  1. 不把私有维护文档推到开源仓库
  2. 不把真实 key、真实证书、SSH、服务器路径写进开源仓库
  3. 不把“只在自己机器可跑”的路径或脚本当作默认说明
  4. 文档改动后要同步复查程序默认值
  5. 推送前同时检查“当前树”和“历史提交”

10.2 开源线更值得继续补充的内容

后面如果继续完善开源版，优先级较高的方向有：

  1. 增加“如何从原始法规/案例文本生成 dense 索引”的公开脚本或说明
  2. 给知识库构建增加更多示例
  3. 增加 issue / PR 模板
  4. 增加更完整的开源部署示例
  5. 增加一套最小 mock 运行方案，降低首次体验门槛

10.3 每次开源推送前建议执行的检查

建议形成固定动作：

  1. 扫描当前树是否有敏感信息
  2. 扫描即将推送的提交历史是否有敏感信息
  3. 检查 README、docs、.env.example、默认配置是否一致
  4. 跑一次前端构建与后端基本启动验证
  5. 检查开源快照标签是否需要同步更新

11. 结论

本次 SafetyRAISE 从部署版整理为开源版，不是一次简单的“复制代码到新仓库”，而是完整做了下面几步：

  1. 裁掉私有维护内容
  2. 脱敏配置与默认值
  3. 重写公开文档
  4. 补齐运行资产说明与知识库样例
  5. 校正文档与程序默认值的一致性
  6. 推送到开源仓库
  7. 发现并修复公开历史中的敏感信息残留
  8. 重新确认开源主线与开源快照安全可用

因此，这份开源版可以作为后续公开展示、继续演进和外部协作的基线；私有部署版则继续承担现网与内部迭代职责，两条线今后应继续分开维护。
