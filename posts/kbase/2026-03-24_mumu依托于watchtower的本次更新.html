全局 Watchtower 自动更新部署与运维归档记录

记录时间： 2026-03-24

核心目标： 实现多台服务器上 Watchtower 的精确卡点更新（北京时间凌晨）、自我更新功能，并绝对保障底层数据库与游戏存档的数据安全。

1. 核心踩坑与通用规则总结

1.1 Docker API 版本兼容问题

  • 现象： 宿主机 Docker 版本较新（29.3.0），而 Watchtower 容器内部自带的客户端默认 API 版本较旧（1.25），导致不断报错 client version 1.25 is too old. Minimum supported API version is 1.40。
  • 解决： 所有 Watchtower 的启动命令必须强制带上 -e DOCKER_API_VERSION=1.44 参数，以提升通信协议版本。

1.2 定时任务（Cron）的语法与时区差异

  • 系统 Cron vs Watchtower Cron： Linux 系统自带的 Cron 是 5 位数（分 时 日 月 周）；而 Watchtower 使用的 robfig/cron 是 6 位数（秒 分 时 日 月 周）。
  • 时区对齐： 为避免在 UTC 服务器上手动换算时间，强制传入环境变量 -e TZ=Asia/Shanghai。传入后，--schedule "0 30 4 *" 即代表精准的北京时间 04:30:00。

1.3 白名单模式（最高安全级别）

为了防止 Watchtower 误伤面板应用（如 1Panel）或强制更新数据库（导致数据表结构与代码脱节），所有服务器均摒弃全局模式，采用白名单指定容器名。

  • 做法： 在启动命令的最后一行，按空格依次列出需要监控的容器名，并且务必包含 watchtower 自身。

2. 各服务器详细部署方案

2.1 MuMuAINovel 服务器 (RackNerd)

  • 监控对象： mumuainovel、watchtower。（坚决排除了 mumuainovel-postgres）。
  • 触发时间： UTC 时间 20:00:00（即北京时间 04:00:00）。
  • 执行命令：

Bash

    docker run -d \
        --name watchtower \
        --restart always \
        -v /var/run/docker.sock:/var/run/docker.sock \
        -e DOCKER_API_VERSION=1.44 \
        containrrr/watchtower \
        --cleanup \
        --schedule "0 0 20 * * *" \
        mumuainovel watchtower

2.2 Minecraft 宝可梦服务器 (高危数据处理)

  • 潜在雷区： 该服务器已配置系统 Cron 在每天北京时间 04:00 (UTC 20:00) 执行 mcctl backup 进行长达数分钟的世界存档备份。若此时 Watchtower 强杀进程，将导致毁档。
  • 防护措施 1（延长宽限期）： 已在 /opt/mc/compose.yml 的 mc 服务下追加 stop_grace_period: 2m，给予服务器 120 秒的时间执行 save-all flush 落盘保存。
  • 防护措施 2（错峰执行）： Watchtower 更新时间延后半小时。
  • 触发时间： 北京时间 04:30:00。
  • 执行命令：

Bash

    docker run -d \
        --name watchtower \
        --restart always \
        -v /var/run/docker.sock:/var/run/docker.sock \
        -e DOCKER_API_VERSION=1.44 \
        -e TZ=Asia/Shanghai \
        containrrr/watchtower \
        --cleanup \
        --schedule "0 30 4 * * *" \
        mc-mc-1 watchtower

2.3 包含 1Panel 面板的服务器

  • 监控对象： firefox（NoVNC 浏览器）、watchtower。避开 1Panel 的所有核心组件。
  • 触发时间： 北京时间 04:00:00。
  • 执行命令：

Bash

        --name watchtower \
        --restart always \
        -v /var/run/docker.sock:/var/run/docker.sock \

        --cleanup \
        --schedule "0 0 4 * * *" \
        firefox watchtower

2.4 Komari 探针服务器

  • 监控对象： komari、watchtower。
  • 触发时间： 北京时间 04:00:00。
  • 执行命令：

Bash

3. 运维脚本 (mumu_ops) 兼容性说明

针对 RackNerd VPS 上的 MuMuAINovel 项目，现有的 mumu_ops 目录下的自动化运维脚本无需任何修改即可完美兼容当前的 Watchtower 白名单模式。

  • 防冲突机制生效中： mumu-full-update.sh 与 mumu-resetdb-upgrade.sh 均内置了 wt_stop() 与 wt_start_back() 拦截函数。当执行手动全量更新或清库升级时，脚本会自动停用名为 watchtower 的容器，并在 trap cleanup EXIT 流程中将其恢复。这彻底杜绝了手动运维与自动更新在同一时刻对容器控制权的争夺。
