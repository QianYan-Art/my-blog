RackNerd VPS 部署 MuMuAINovel 全流程笔记

2025-12-1114 min
e-RackNerd
机器环境：RackNerd VPS（2 vCPU / 2.5G RAM / 40G SSD）+ 额外 1.2G 日志盘挂载到 /mnt/logs
系统：Ubuntu 22.04
项目目录：/root/MuMuAINovel
访问架构：浏览器 → EdgeOne（域名 + 免费证书 + 缓存）→ VPS:80 (Nginx) → 127.0.0.1:8000 (MuMuAINovel 容器)

---

一、VPS 基础环境 & 日志盘

1.1 查看磁盘与日志盘

df -h | grep /mnt/logs
ls -ld /mnt/logs

确认：

  • 日志盘已经挂载到 /mnt/logs
  • 大小约 1.2G

为 MuMuAINovel 创建独立日志目录：

mkdir -p /mnt/logs/mumuainovel

1.2 Swap / 系统初始化（略要点）

  • 为 2.5G 内存机配置了 4G 左右的 swap，用于缓冲镜像拉取与容器运行时的内存峰值。
  • 调整了 swappiness 等参数，以避免频繁 swap 对性能影响过大。

（详细 Swap 配置步骤保留在单独的初始化文档里，此处不展开。）

---

二、安装 Docker & Docker Compose

使用官方脚本安装 Docker：

cd /root
curl -fsSL https://get.docker.com -o get-docker.sh
sh get-docker.sh

校验安装结果：

docker -v
# 新版本使用 docker compose 子命令
docker compose version
备注：本机没有单独的 docker-compose 可执行文件，使用的是 docker compose 子命令形式。

---

三、克隆 MuMuAINovel 项目

cd /root
git clone https://github.com/xiamuceer-j/MuMuAINovel.git
cd MuMuAINovel

项目根目录即：/root/MuMuAINovel。

---

四、配置 .env 环境变量

根据官方文档，先复制示例 env 文件：

cp backend/.env.example .env
nano .env

核心配置内容（示例）：

# =====================
# 数据库配置
# =====================
POSTGRES_DB=mumuai_novel
POSTGRES_USER=mumuai
POSTGRES_PASSWORD=自己设定的强密码

# =====================
# OpenAI / 中转配置
# =====================
OPENAI_API_KEY=你的真实 key
OPENAI_BASE_URL=https://juya.owl.ci/v1

# =====================
# 默认 AI 配置
# =====================
DEFAULT_AI_PROVIDER=openai
DEFAULT_MODEL=gpt-4o-mini
DEFAULT_TEMPERATURE=0.7
DEFAULT_MAX_TOKENS=2000

# =====================
# 本地账号登录
# =====================
LOCAL_AUTH_ENABLED=true
LOCAL_AUTH_USERNAME=admin
LOCAL_AUTH_PASSWORD=自己设定的密码

# =====================
# 端口与时区
# =====================
APP_PORT=8000
TZ=Asia/Tokyo
注意：

>

- OPENAI_BASE_URL 配置为中转地址（如 juya）。
- DEFAULT_AI_PROVIDER 不要留空，典型为 openai。
- API KEY 属于敏感信息，外泄要立刻废弃重建。

五、修改 docker-compose.yml

5.1 使用官方镜像而非本地构建

原始配置中可能包含：

mumuainovel:
  build:
    context: .
    dockerfile: Dockerfile
  image: mumujie/mumuainovel:latest

在内存较小的 VPS 上，为避免本地构建消耗资源，将其改为仅使用镜像：

services:
  mumuainovel:
    image: mumujie/mumuainovel:latest
    container_name: mumuainovel
    # build 段直接注释掉或删除
    # build:
    #   context: .
    #   dockerfile: Dockerfile

Postgres 服务：

  postgres:
    image: postgres:18-alpine
    container_name: mumuainovel-postgres
    environment:
      - POSTGRES_DB=${POSTGRES_DB}
      - POSTGRES_USER=${POSTGRES_USER}
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
    ports:
      - "0.0.0.0:5432:5432"
    volumes:
      - ./postgres-data:/var/lib/postgresql/data
备注：Postgres 暴露 5432 到外网是方便调试用，生产环境建议限制访问来源或关闭端口映射。

5.2 日志挂载到独立日志盘

在 mumuainovel 服务下配置 volumes：

    volumes:
      - /mnt/logs/mumuainovel:/app/logs
      - ./.env:/app/.env:ro
  • /mnt/logs/mumuainovel：宿主机日志目录（在 1.2G 日志盘上）
  • /app/logs：容器内应用日志目录
  • .env 只读挂载到 /app/.env

5.3 端口仅绑定到本地 127.0.0.1

为配合 Nginx 反向代理，将容器端口绑定到本机：

    ports:
      - "127.0.0.1:8000:8000"

这样：

  • 宿主机本地可以通过 127.0.0.1:8000 访问应用
  • 外网不能直接访问 8000 端口，只能通过 Nginx（80 端口）+ 域名转发

5.4 AI 相关环境变量

在 environment 段注入配置：

    environment:
      - OPENAI_API_KEY=${OPENAI_API_KEY:-}
      - OPENAI_BASE_URL=${OPENAI_BASE_URL:-https://api.openai.com/v1}
      - GEMINI_API_KEY=${GEMINI_API_KEY:-}
      - GEMINI_BASE_URL=${GEMINI_BASE_URL:-}
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY:-}
      - ANTHROPIC_BASE_URL=${ANTHROPIC_BASE_URL:-}
      - DEFAULT_AI_PROVIDER=${DEFAULT_AI_PROVIDER:-openai}
      - DEFAULT_MODEL=${DEFAULT_MODEL:-gpt-4o-mini}
      - DEFAULT_TEMPERATURE=${DEFAULT_TEMPERATURE:-0.7}
      - DEFAULT_MAX_TOKENS=${DEFAULT_MAX_TOKENS:-2000}
实际部署中，通过 .env 明确写入 DEFAULT_*，避免被宿主机环境变量覆盖引发“配置改不动”的错觉。

六、拉取镜像并启动容器

6.1 预拉镜像（可选）

cd /root/MuMuAINovel

docker pull mumujie/mumuainovel:latest
docker pull postgres:18-alpine

6.2 启动服务

使用 docker compose 子命令：

cd /root/MuMuAINovel

docker compose up -d

查看容器状态：

docker compose ps

典型输出：

NAME                   IMAGE                        COMMAND                  SERVICE       CREATED         STATUS                   PORTS
mumuainovel            mumujie/mumuainovel:latest   "python -m uvicorn a…"   mumuainovel   ...             Up (healthy)            127.0.0.1:8000->8000/tcp
mumuainovel-postgres   postgres:18-alpine           "docker-entrypoint.s…"   postgres      ...             Up (healthy)            0.0.0.0:5432->5432/tcp

6.3 检查日志目录

如需清空某个日志文件：

七、Nginx 反向代理配置

目标：

  • 外部只访问域名 + 80 / 443
  • Nginx 转发到本地 127.0.0.1:8000

7.1 安装 Nginx

apt update
apt install -y nginx

7.2 创建站点配置文件

新建 /etc/nginx/sites-available/mumuainovel.conf：

nano /etc/nginx/sites-available/mumuainovel.conf

写入以下内容（示例域名 yourdomain.com）：

server {
    listen 80;
    listen [::]:80;
    server_name yourdomain.com;
    # 或者 novel.yourdomain.com 等子域名

    location / {
        proxy_pass         http://127.0.0.1:8000;
        proxy_http_version 1.1;

        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;

        proxy_set_header   Upgrade           $http_upgrade;
        proxy_set_header   Connection        "upgrade";
    }
}

启用站点并关闭默认站点：

ln -s /etc/nginx/sites-available/mumuainovel.conf \
      /etc/nginx/sites-enabled/mumuainovel.conf

rm /etc/nginx/sites-enabled/default

检查配置并重启 Nginx：

nginx -t
systemctl restart nginx
systemctl enable nginx

此时可在本地浏览器直接访问：

http://74.48.40.123

如果能正常打开 MuMuAINovel 页面，说明 Nginx ↔ Docker 部分链路无误。

八、EdgeOne 域名与 HTTPS 配置

8.1 DNS 解析

在 EdgeOne 控制台：

  • 添加 A 记录：

- 主机记录：@ 或 novel（对应根域或子域）

- 记录值：74.48.40.123

- 代理/加速：开启（走 EdgeOne 节点）

等待解析生效（通常 1–5 分钟）。

8.2 免费证书与回源配置

在 EdgeOne：

  1. 申请 / 启用针对该域名的免费 SSL 证书。
  1. 对应域名的加速配置中：

- 客户端到 EdgeOne：启用 HTTPS + 绑定免费证书

- EdgeOne 回源：使用 HTTP

- 回源地址：74.48.40.123

- 回源端口：80

最终链路：

  • 浏览器：https://yourdomain.com
  • EdgeOne：TLS 终止 → HTTP 回源 → http://74.48.40.123:80
  • Nginx：80 → 反代 → 127.0.0.1:8000
  • MuMuAINovel 容器：监听 8000

九、一键更新脚本

在 /root 下创建更新脚本：

nano /root/update_mumu.sh

内容示例：

#!/usr/bin/env bash
# 一键更新 MuMuAINovel（代码 + 镜像 + 重启容器）

set -e

PROJECT_DIR="/root/MuMuAINovel"

echo "==> 进入项目目录：$PROJECT_DIR"
cd "$PROJECT_DIR"

echo "==> 拉取最新代码（git pull）..."
git pull || echo "git pull 失败（可能是本地有改动），跳过。"

echo "==> 检测 docker compose 命令..."
if docker compose version &>/dev/null; then
  COMPOSE="docker compose"
else
  echo "❌ 未找到 docker compose 命令，请确认 Docker 已正确安装。"
  exit 1
fi

echo "==> 使用命令：$COMPOSE"

echo "==> 拉取最新镜像..."
$COMPOSE pull

echo "==> 重新启动容器（up -d）..."
$COMPOSE up -d

echo "==> 清理无用镜像..."
docker image prune -f

echo "✅ MuMuAINovel 更新完成，当前容器状态："
$COMPOSE ps

赋予执行权限：

chmod +x /root/update_mumu.sh

以后更新时只需：

bash /root/update_mumu.sh

十、配置“没生效”的排错记录

部署过程中遇到的典型现象：

  • 修改 .env 后，重启容器 / 重拉镜像，Web 端看到的模型配置似乎仍是旧的。
  • 实际排查后发现：

- 容器内 env 中的变量值已经更新（例如 DEFAULT_MODEL 改为新模型名）。

- 真正导致“看起来没变”的原因是 EdgeOne 缓存了前端页面/接口响应。

10.1 检查容器内生效的环境变量

# 查看容器内的环境变量
cd /root/MuMuAINovel

docker compose ps

docker exec -it mumuainovel env | grep -E "OPENAI_BASE_URL|DEFAULT_AI_PROVIDER|DEFAULT_MODEL"

如果这里显示的已经是新值，则说明后端配置其实已经更新。

10.2 EdgeOne 缓存导致的假“旧配置”

  • 由于 EdgeOne 在边缘节点缓存了部分前端资源 / API 返回，浏览器访问时得到的是缓存内容，看起来像“配置没保存/没生效”。
  • 解决方式：

- 在 EdgeOne 控制台中针对该路径 / 域名清除缓存；或

- 临时关闭加速 / 缓存，然后再次访问；

- 本地浏览器也可尝试 Ctrl+F5 强制刷新，或在无痕模式访问。

确认方式：

  • 直接通过 VPS IP + 80 端口访问 Nginx（绕过 EdgeOne）：
http://74.48.40.123
  • 对比域名访问 https://yourdomain.com 的表现，若 IP 访问看到的是新配置，但域名访问仍是旧配置，则基本可确认是边缘缓存问题。

十二、小结

这次在 RackNerd 小内存 VPS 上部署 MuMuAINovel 的完整流程包括：

  1. 初始化系统与 Swap，并挂载独立日志盘到 /mnt/logs；
  1. 安装 Docker 与 docker compose；
  1. 克隆 MuMuAINovel 项目到 /root/MuMuAINovel；
  1. 配置 .env 包含数据库、中转 OpenAI、默认模型及本地账号；
  1. 调整 docker-compose.yml：使用官方镜像、挂载日志到 /mnt/logs/mumuainovel、只绑定本地 8000 端口；
  1. 启动 postgres 与 mumuainovel 容器，确认应用可在本地访问；
  1. 安装 Nginx，配置 80 端口反向代理到 127.0.0.1:8000；
  1. 在 EdgeOne 上完成域名解析、免费证书和 HTTPS 回源配置；
  1. 编写一键更新脚本，简化后续维护操作；
  1. 通过容器环境变量检查 + 绕过 EdgeOne 访问，定位并解决了“配置似乎没有更新”这一缓存问题。

这份笔记可以作为今后在其他 VPS 上复用 / 再部署 MuMuAINovel 的模板，也方便排查类似“配置不生效”“日志占满磁盘”“域名访问异常”等常见问题。
