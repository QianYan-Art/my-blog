本机运行时与家目录依赖体检记录 - 2026-05-15

一、检查目标

本次检查主要解决以下问题：

  1. 明确当前本机常用运行时、包管理器、CLI 工具的真实安装来源与命中路径。
  2. 判断 C:\Users\lenovo 家目录是否已经被当成 Node/Bun 项目目录使用。
  3. 核对 ohmypi（实际包名为 @oh-my-pi/pi-coding-agent）当前依赖是否完整、入口是否正常。
  4. 找出环境中的主要风险点，尤其是多版本并存、路径优先级冲突、家目录污染问题。

二、已确认的关键结论

1. PowerShell 默认启动目录已调整

Windows Terminal 中 PowerShell 7 (Scoop) 的 startingDirectory 已从 %USERPROFILE% 改为：

C:\tmp\workplace

这样做的目的：

  1. 避免每次打开终端都落到家目录。
  2. 避免在家目录误执行 npm install、bun add 等命令。
  3. 避免 starship 因家目录存在 package.json / bun.lock 而持续显示 Bun 模块。

2. 家目录已被当成 JS/Bun 项目目录使用

C:\Users\lenovo 当前存在以下关键文件或目录：

package.json
package-lock.json
bun.lock
node_modules

这说明此前曾经直接在家目录执行过 Node/Bun 项目相关命令，而不是纯粹的“安装运行时”。

家目录顶层 package.json 当前内容对应的直接依赖为：

dependencies:
- @oh-my-pi/pi-coding-agent ^15.0.2
- glob ^11.1.0
- rimraf ^6.1.3

devDependencies:
- eslint ^9.39.4

结论：

  1. 家目录现在不能再看成“纯净用户目录”。
  2. 它实际上已经承担了一个 Bun/Node 项目目录的角色。
  3. 后续若继续在家目录执行 npm install / bun install / bun add，会继续污染该目录。

3. ohmypi 当前状态正常

实际安装内容核对结果如下：

运行时:
- bun = C:\Users\lenovo\.bun\bin\bun.exe
- bun 版本 = 1.3.14

CLI 入口:
- omp = C:\Users\lenovo\.bun\bin\omp.exe

实际包:
- @oh-my-pi/pi-coding-agent@15.0.2

@oh-my-pi/pi-coding-agent 包信息中声明：

engines.bun >= 1.3.14
bin.omp -> src/cli.ts

结论：

  1. 当前 Bun 版本满足 ohmypi 的最低要求。
  2. omp 命令入口存在且链路正常。
  3. 当前未发现 ohmypi 因依赖缺失或 Bun 版本不足而失效。

四、重要冲突与风险点

风险 1：家目录项目化

这是当前最明确、最真实的问题源头。

表现：

  1. 家目录里存在 package.json、bun.lock、package-lock.json、node_modules。
  2. 家目录一度被 npm 和 bun 都操作过。
  3. starship 会把家目录识别成 Bun/Node 项目。

风险：

  1. 容易误把“项目安装”当成“全局安装”。
  2. 容易继续把依赖装到 C:\Users\lenovo。
  3. 后续清理难度会逐步上升。

风险 2：同名命令存在多份来源

实际 where 检查中，以下命令存在多路径并存情况：

python
node
npm
codex
java
git

其中最值得注意的是：

#### codex

存在以下多份命中：

D:\Develop\nodejs\node_global\codex.cmd
C:\Users\lenovo\AppData\Local\OpenAI\Codex\bin\codex.exe
C:\Program Files\WindowsApps\OpenAI.Codex_...\codex.exe

当前 shell 命中的是 npm 全局安装版 codex，不是桌面安装目录附带的那份。

风险：

  1. 升级来源容易混淆。
  2. 不同入口将来可能版本不一致。

#### python

存在 JetBrains Python、MSYS2 Python、Miniconda Python、WindowsApps 占位项。

风险：

  1. 某些脚本若不固定解释器，可能走错环境。
  2. 虚拟环境、系统环境、conda 环境之间可能互相污染。

风险 3：Node 运行时与 npm 全局包分离

当前结构：

Node 运行时 -> D:\jetbrains\node\nodejs
npm 全局前缀 -> D:\Develop\nodejs\node_global

风险：

  1. 维护时需要同时记住两个目录。
  2. 若以后迁移 Node，可能需要重新确认 npm 全局 prefix 与 PATH。

风险 4：家目录同时存在 package-lock.json 与 bun.lock

说明同一个目录先后被两套生态工具处理过：

  1. npm 侧留下 package-lock.json
  2. bun 侧留下 bun.lock

风险：

  1. 以后继续在家目录混用 npm install 与 bun install，会进一步制造依赖漂移。
  2. 锁文件与真实安装树可能越来越难解释。

五、当前判断：哪些地方是正常的，哪些地方不能再继续放任

当前可接受、暂不需要立即处理

  1. ohmypi 当前安装链路正常。
  2. bun 当前版本满足 ohmypi 需求。
  3. git、docker、kubectl、mvn、java、uv 均能正常命中。
  4. Bun 升级后留下 bun.exe.outdated 不属于严重故障。

当前最不该继续放任的问题

  1. 再次在 C:\Users\lenovo 直接执行 npm install / bun add / bun install
  2. 不区分 npm 全局安装和项目安装
  3. 在不明确当前命中路径的前提下随意升级 codex / python / node

六、后续建议

1. 使用纪律

以后执行 Node/Bun 相关安装命令时，先确认两件事：

  1. 当前目录是否正确
  2. 这次安装是“项目依赖”还是“全局工具”

建议心法：

装工具 -> 先确认是不是全局
装依赖 -> 先 cd 到项目目录
当前目录如果是 ~，默认提高警惕

2. 近期不建议直接清理家目录这套依赖

原因：

  1. 当前 ohmypi 正在依赖这套目录结构。
  2. 贸然移动或删除，容易造成 CLI 失效。
  3. 目前最优策略是“先止血，再审计，再迁移”。

3. 建议后续单独做一次“家目录 JS/Bun 项目迁移评估”

目标不是立刻迁移，而是先搞清：

  1. 家目录 node_modules 中哪些是 ohmypi 必需依赖
  2. 是否还混入了其他历史项目依赖
  3. 如果将来把 ohmypi 迁到单独目录，应如何保持 omp 继续可用

七、一句话总结

本机环境当前可以正常使用，但已经形成“多来源运行时 + 多路径命中 + 家目录项目化”的状态。 短期内最正确的策略不是大清理，而是：

  1. 继续使用新的终端默认目录 C:\tmp\workplace
  2. 不再在家目录执行项目安装命令
  3. 后续再对家目录这套 Node/Bun 依赖做单独迁移评估
