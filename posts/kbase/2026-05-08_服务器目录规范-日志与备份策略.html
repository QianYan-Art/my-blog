服务器目录规范、日志与备份策略

2026-05-085 min

2026-05-08 服务器目录规范、日志与备份策略

1. 背景

这台服务器后续会承载多个站点。 当前磁盘结构只有一个主要根分区：

  • /dev/vda4 -> /

因此本次没有做“物理分区重切”，而是改成更适合现阶段的：

  • 统一目录规范
  • 日志统一收口
  • 自动轮转与自动备份

这套方案的优点是：

  • 不需要停机重分区
  • 对新站点扩展友好
  • 维护成本低
  • 后期如果确实需要再迁移到独立盘或对象存储也方便

2. 目录规范

2.1 应用编排目录

  • /srv/apps

用途：

  • 存放 Docker Compose 文件
  • 存放应用 .env
  • 存放站点说明文件

当前已有：

  • /srv/apps/zipline
  • /srv/apps/SITES.md

2.2 业务数据目录

  • /srv/data

用途：

  • 存放上传文件
  • 存放应用持久化数据
  • 存放数据库数据卷

当前已有：

  • /srv/data/zipline

2.3 日志目录

  • /srv/logs
  • /srv/logs/nginx

用途：

  • 统一管理 Nginx 站点访问日志和错误日志

2.4 备份目录

  • /srv/backups

用途：

  • 存放每日自动备份产物

3. Nginx 日志统一收口

3.1 当前站点日志

Komari：

  • /srv/logs/nginx/qianyan.access.log
  • /srv/logs/nginx/qianyan.error.log

Zipline 图床：

  • /srv/logs/nginx/img.access.log
  • /srv/logs/nginx/img.error.log

3.2 站点日志轮转

新增 logrotate 配置：

  • /etc/logrotate.d/srv-nginx-sites

策略：

  • 每天轮转
  • 保留 14 份
  • 自动压缩
  • 使用 USR1 通知 Nginx 平滑切日志

4. journald 限额

新增配置文件：

  • /etc/systemd/journald.conf.d/99-qianyan-limits.conf

内容：

  • SystemMaxUse=500M
  • RuntimeMaxUse=100M
  • MaxRetentionSec=14day
  • Compress=yes

作用：

  • 防止系统日志无限增长
  • 保持近 14 天可查

5. Docker 日志轮转

Docker 守护进程配置：

  • /etc/docker/daemon.json

当前策略：

{
  "live-restore": true,
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "20m",
    "max-file": "5"
  }
}

作用：

  • 每个容器日志最多约 100MB
  • 防止容器 stdout/stderr 把磁盘吃满

6. 自动备份

6.1 备份脚本

  • /usr/local/bin/qianyan-backup.sh

6.2 systemd 服务与定时器

  • /etc/systemd/system/qianyan-backup.service
  • /etc/systemd/system/qianyan-backup.timer

6.3 执行时间

  • 每天 04:20

6.4 保留策略

  • 自动保留最近 7 天目录备份
  • 超过 7 天的旧备份自动删除

6.5 备份内容

脚本当前会备份：

  • /srv/apps
  • /srv/data
  • /opt/komari
  • /etc/nginx
  • /etc/letsencrypt
  • /etc/ssh/sshd_config
  • /etc/ssh/sshd_config.d
  • /etc/systemd/journald.conf.d
  • /etc/logrotate.d
  • /etc/docker/daemon.json
  • zipline.sql 数据库导出

6.6 备份目录结构

示例：

/srv/backups/2026-05-08_06-42-36
/srv/backups/latest -> /srv/backups/2026-05-08_06-42-36

6.7 当前验证状态

备份定时器当前已验证可用：

  • 定时器：qianyan-backup.timer
  • 执行时间：每天 04:20
  • 已手动触发过一次
  • latest 软链接正常更新

7. 最近一次清理记录

后续又做过一次保守清理。

清理内容：

  • 删除远端 /tmp 中的部署临时脚本、调试脚本、临时 .tmp.js / .tmp.ico
  • 清理 apt 缓存
  • 执行 Docker 镜像与构建缓存清理

清理原则：

  • 不删除业务数据
  • 不删除证书
  • 不删除容器持久化目录
  • 不删除实际站点日志

清理后状态：

  • APT 缓存约 12K
  • Docker 无可回收镜像
  • 根分区使用约 8.6G / 157G

8. 站点清单文件

服务器内新增：

  • /srv/apps/SITES.md

用途：

  • 记录每个域名对应的应用
  • 记录反向代理端口
  • 记录数据目录
  • 方便以后新增站点时快速定位

9. 当前建议的新增站点流程

以后每新建一个站点，建议固定按下面做：

  1. 在 /srv/apps/<site> 放 Compose 或应用配置
  2. 在 /srv/data/<site> 放持久化数据
  3. Nginx 站点文件单独放 /etc/nginx/sites-available/<domain>
  4. 日志写到 /srv/logs/nginx/<site>.access.log 和 <site>.error.log
  5. 用 certbot --nginx -d <domain> 签证书
  6. 备份脚本默认会把整套目录带上，无需单独改逻辑

10. 当前策略的边界

这套方案已经足够支撑：

  • 多个 Docker 站点
  • 多个 Nginx 反代站点
  • 图床 / 面板 / API / 静态站

但它不是物理隔离方案。 如果未来出现以下情况，再考虑拆分磁盘或对象存储：

  • 图床上传量很大
  • 日志量很大
  • 视频类文件明显增加
  • 想把上传文件迁移到 S3 / R2 / OSS

11. 结论

当前服务器已经具备：

  • 统一目录结构
  • 统一日志入口
  • 自动轮转
  • 自动证书续期
  • 自动每日备份

对“建各种站”的用途来说，这已经是一个比较稳的第一版基线。 后续新增站点时，尽量不要再各自散放目录，而是继续沿用这套结构。
