NetCup 上 CLIProxyAPI 迁移与 EdgeOne 回源配置记录

2026-05-1716 min
f-NetCup

1. 目标与结论

本次目标：

  • 放弃旧 JP NAT 机上那套专门为 Codex + suyuan 临时修补过很多次的运行方式。
  • 将 cliproxyapi 迁移到 NetCup 建站机，优先追求结构整洁、长期稳定、少开额外端口。
  • 保留原本对图片生成长耗时请求的保活能力。
  • 不再继续叠加 newapi，避免无意义增加一层网关和维护复杂度。

最终结论：

  • 不部署 newapi。
  • 继续沿用 EdgeOne 加速域名 suyuan.4071253.xyz。
  • EdgeOne 回源到新机 80 端口，由新机 nginx 反代到本机 cliproxyapi。
  • 不再对公网开放 25555 之类额外端口。

2. 为什么不部署 newapi

本次实际需求是：

  • 反代 cliproxyapi
  • 保留 Gemini / Vertex / Claude / OpenAI 这套现有认证与模型路由
  • 处理图片生成时的长连接首字节超时问题

newapi 的价值更偏向：

  • 通用 OpenAI 兼容聚合网关
  • 多上游统一计费、额度、模型映射
  • 适合“面向大量 API key 用户”的平台化场景

但这次并不需要它来解决核心问题。当前目标是把你已经可用的 cliproxyapi 稳定搬家，而不是再引入一个新的控制层。

因此本次明确采用：

  • cliproxyapi 作为唯一核心服务
  • edge-proxy 仅负责长耗时图片请求保活
  • nginx 仅负责域名入口反代

5. 本次保留的有效改动

5.1 保留：图片生成长耗时保活

保留文件：

  • /root/cliproxyapi-edge-proxy/cliproxyapi_edge_proxy.py

保留逻辑：

  • 对以下路径的 POST 请求做长耗时保活：

    • - /v1/images/generations - /v1/images/edits - /v1/images/variations

  • 若上游在短时间内未返回首字节，先回 200
  • 后续定期写空格 keepalive
  • 等待上游真正完成后，再把实际响应流继续转发

这是本次必须保留的关键改动，因为它直接对应你原本“生图容易超时”的真实需求。

目的：

  • 保留会话粘性

  • 保留 vertex 模型前缀区分能力

5.3 去除：SSE filter

没有迁移：

  • 这层是为了旧阶段 Codex 对 SSE 中非标准字段的严格解析做的专项修补

10. 当前遗留事项

10.1 OpenAI / Codex 某个旧认证存在 refresh token 复用告警

日志中存在：

  • refresh_token_reused

含义：

  • 某个旧 OpenAI/Codex 认证文件中的 refresh token 已失效或已在别处被刷新过

当前影响：

  • 不影响服务启动
  • 不影响 /v1/models 正常响应
  • 但后续若你仍要让某条 OpenAI/Codex OAuth 账号真正可用，需要重新登录一次该账号

处理方式：

  • 在新机 cliproxyapi 目录下重新执行对应登录流程
  • 或替换掉失效的那份 auth 文件

10.2 直接访问裸 IP 会出现 404 属于正常现象

例如：

  • http://213.109.161.23/v1/models

若没有正确 Host: suyuan.4071253.xyz，命中的是默认站点，不是该反代站点，因此返回 404 是正常的。

这不代表 cliproxyapi 有故障。

11. 通过旧 JP 节点作为新机出站代理

为避免新机直接以自身机房 IP 对外访问上游，同时保留当前建站机的整洁入口结构，本次又增加了一层“仅本机可见”的出站代理：

新机 cliproxyapi
  -> proxy-url = socks5://127.0.0.1:24080
  -> 新机 sing-box 本地 socks 入站
  -> sing-box Shadowsocks 出站
  -> 旧 JP NAT 节点 SS 服务
  -> 上游服务

11.1 新机 sing-box 新增内容

新增配置分片：

  • /etc/sing-box/conf/local-socks-inbound.json
  • /etc/sing-box/conf/oldjp-ss-outbound.json
  • /etc/sing-box/conf/route-cliproxyapi-proxy.json

作用：

  • 127.0.0.1:24080 提供本地 socks 入口
  • 出站通过旧机 nat-jp-3h8e.aproxy.top:28005
  • 仅本机可用，不增加新的公网监听端口

11.2 cliproxyapi 配置变更

/root/cliproxyapi/config.yaml 中已改为：

  • proxy-url: "socks5://127.0.0.1:24080"

同时 claude-api-key 条目下也同步指定了：

  • proxy-url: "socks5://127.0.0.1:24080"

这样 cliproxyapi 的上游出站将优先走旧 JP 节点代理，而不是直接用 NetCup 机房 IP。

11.3 实测结果

已实测：

  • 新机 sing-box 本地监听 127.0.0.1:24080
  • 新机通过该 socks 访问 https://api.ipify.org
  • 返回出口 IP：60.67.40.243

说明：

  • 新机实际已通过旧 JP 节点出站
  • 同时 https://suyuan.4071253.xyz/v1/models 仍然返回 200

11.4 严格主备代理策略

最终又改为“严格主备”，不再使用 urltest 自动优选。

当前逻辑：

  • cliproxyapi 正常情况下固定走旧 JP 节点 oldjp-ss-out
  • 仅当健康检查确认旧 JP SS 不可用时，才把主路由切到 direct
  • 旧 JP SS 恢复后，会自动切回 oldjp-ss-out

新增与调整：

  • /etc/sing-box/conf/local-socks-inbound.json

    • - 127.0.0.1:24080：cliproxyapi 实际使用的本地 socks - 127.0.0.1:24081：仅供健康检查脚本验证旧 JP SS 是否可用

  • /etc/sing-box/conf/route-cliproxyapi-proxy.json

    • - 默认将 cliproxyapi-local-socks 路由到 oldjp-ss-out - 健康检查口固定路由到 oldjp-ss-out

  • /root/check-cliproxyapi-proxy.py

    • - 通过 127.0.0.1:24081 检测旧 JP SS - 若不可用，则把 cliproxyapi-local-socks 切到 direct - 若恢复可用，则切回 oldjp-ss-out

  • /etc/systemd/system/cliproxyapi-proxy-failover.service
  • /etc/systemd/system/cliproxyapi-proxy-failover.timer

定时检查周期：

  • OnBootSec=45s
  • OnUnitActiveSec=30s

当前状态记录文件：

  • /var/lib/cliproxyapi-proxy-mode

当前实测：

  • curl --socks5-hostname 127.0.0.1:24080 https://api.ipify.org
  • 返回出口 IP：126.40.18.15

说明：

  • 现在从 suyuan.4071253.xyz 进入后，cliproxyapi 的上游出口默认是旧 JP 节点代理
  • 只有旧 JP 代理真正不可达时，才会短暂回退到新机公网 213.109.161.23

12.3 Vertex 与 manual 前缀修复记录

Vertex service account 导入后的 location 修复

问题现象：

  • Vertex 原始 service account JSON 本身可用
  • 但通过管理页导入后，生成的 auth 文件被写成：

    • - location: "asia-northeast1"

  • 调用 vtx/gemini-3.1-pro-preview 时会报：

    • - Publisher Model ... was not found or your project does not have access to it

最终修复：

  • 将 /root/cliproxyapi/auths/vertex-project-74ecad93-9ffc-48c4-a8d.json 中的：

    • - location: "asia-northeast1"

  • 改为：

    • - location: "global"

结论：

  • 对这份 Vertex 配置，gemini-3.1-pro-preview 应使用 global
  • 若后续再次通过管理页导入后又被改回区域节点，需要再次检查该字段

第三方模型 manual/ 前缀恢复

问题现象：

  • 迁移后 manual/ 前缀模型从 /v1/models 中消失

根因：

  • 当前正式配置 /root/cliproxyapi/config.yaml 中，以下两项的 prefix: "manual" 在迁移后丢失：

    • - claude-api-key - codex-api-key

修复方式：

  • 参考迁移时留下的旧快照 /root/config.with-oldjp-ss.yaml
  • 将上述两项的 prefix: "manual" 补回当前正式配置

恢复结果：

  • /v1/models 中重新出现：

    • - manual/gpt- - manual/gpt-image-2 - manual/claude-
