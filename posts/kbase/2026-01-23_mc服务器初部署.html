mc服务器初部署

适用场景：你在日本服务器（Debian 11）上，用 Docker Compose 部署 Fabric 服务器，并把整合包 mods/config 同步到服务端。本文是“今天实际做过的操作”的复盘 + 后续维护指北。

0. 你现在这台服务器上有什么（现状概览）

系统与 Docker

  • 系统：Debian GNU/Linux 11（bullseye）
  • Docker：20.10.x
  • Docker Compose：v5.x

容器 / 镜像（从你截图里看到的）

  1. 游戏服容器
  • 容器：mc-mc-1
  • 镜像：itzg/minecraft-server:stable-java21-jdk
  • 端口：0.0.0.0:25565->25565/tcp
  • 状态：healthy（健康）
  1. 皎月连 / NatPierce 容器
  • 容器：natpierce
  • 镜像：xiyu505/natpierce:latest
  • 状态：Exited (137)（目前未运行）
  • 作用：用于把“虚拟网段/内网穿透/点对点”之类的连接能力带到服务器上（但你现在已经改成公网直连为主，所以它可以不用常驻）。

网络 / 卷

  • Compose 默认会创建一个项目网络：mc_default（给 mc 项目容器互联用）
  • 另外你系统里还有默认的 bridge/host/none
  • natpierce_data 卷：这是皎月连容器的持久化数据卷（不是 MC 的世界存档）

1. 今天你做了哪些关键步骤（以及每步的意义）

1.1 发现系统环境和包安装差异

你一开始以为是 Ubuntu 24.04，但实际是 Debian 11；同时 apt 源里出现 archive 源/安全源的组合。

  • 意义：不同发行版的 Java 包名/版本不同，导致你最开始安装 openjdk-21-jre-headless 报“没有候选”。
  • 解决思路：改用 Docker 方案（容器里自带 Java），避免在宿主机折腾 JDK 版本。

1.2 选择 Docker 方案部署 Minecraft Server（核心架构）

你最终用了 itzg/minecraft-server 这个成熟镜像，并用 docker compose up -d 让它在后台运行。

  • 意义：

- Minecraft 服务端（Fabric/Forge/Paper 等）在容器里跑；

- 世界存档、配置、mods 通过挂载目录持久化；

- 以后更新/重启/备份都变得可控。

1.3 Fabric 服务器启动与首启下载

日志里能看到：

  • 下载 Fabric server jar（对应 MC 1.21.1 + loader 0.18.x）
  • 首次生成 server.properties
  • 监听 *:25565
  • 启动 RCON（25575）

意义：

  • 25565 是玩家进服端口
  • RCON 25575 是“远程控制台端口”，用来让脚本/工具执行服务器命令（白名单、op、say、stop 等）

1.4 把整合包内容同步到服务端（mods/config）

你提供的整合包 zip 是 Modrinth 格式（有 modrinth.index.json + overrides/）。

  • 你最初解压时发现 overrides/mods 只有 1 个 jar（pokecoin），这其实是 Modrinth 格式的正常现象：

- 真正的 mod 列表在 modrinth.index.json

- 客户端启动器会根据 index 去下载依赖

- overrides/ 里只放“额外覆盖文件”（配置、资源包、少量手动 mod）

后来你把现成的 191 个服务端 mods 放进了：

  • /opt/mc/data/mods

意义：

  • 服务端是否能进服、mod 是否生效，关键就看 /opt/mc/data/mods 和 /opt/mc/data/config。

1.5 处理“奇怪文件名导致 NoSuchFileException”

你遇到过 Fabric 扫描 mod 报错：

  • NoSuchFileException: /data/mods/[一些乱码]xxx.jar

这类问题通常来自：

  • 文件名包含非 UTF-8 字符 / 控制字符
  • 复制/解压过程中编码损坏

你后面通过重新同步 mods（并验证目录内 jar 文件名正常）解决了，服务器能完整启动到 Done。

1.6 白名单（whitelist）与 RCON 管理

你启用了白名单，并用 mcctl whitelist-list 查看过（显示为空）。

  • 意义：

- 开服端口在公网时，白名单是第一道保护（防陌生人乱连）

- RCON 让脚本可以自动添加/删除白名单、OP 等

1.7 内存从 8G 调到 10G

你改了 compose 后重启，进程参数显示：

  • -Xmx10G -Xms10G

意义：

  • Xms 初始堆内存，Xmx 最大堆内存
  • 模组服会吃内存，适度加大能减少卡顿/GC 抖动

2. 你的“游戏服目录结构”应该怎么理解

以你现在的部署习惯（/opt/mc）为基准：

  • /opt/mc/compose.yml

- Docker Compose 配置，决定使用哪个镜像、端口映射、环境变量、挂载目录等

  • /opt/mc/.env

- 放一些敏感变量（例如 RCON_PASSWORD）

  • /opt/mc/data/（最重要：持久化数据）

- world/：主世界存档（还有 nether/end 相关目录）

- mods/：服务端 mods（你现在约 191 个）

- config/：模组配置

- server.properties：服务器核心配置

- whitelist.json：白名单列表

- ops.json：管理员列表

- logs/：服务端日志

容器内对应路径一般是 /data/（通过 volume 映射到宿主机 /opt/mc/data/）。

3. 如何查看 / 修改关键服务器设置（server.properties）

3.1 查看 online-mode / whitelist

在宿主机执行：

  • 查看 online-mode：

- grep -E '^online-mode=' /opt/mc/data/server.properties || true

  • 查看白名单是否开启：

- grep -E '^(white-list|enforce-whitelist)=' /opt/mc/data/server.properties || true

说明：

  • online-mode=true：正版认证（玩家名与账号绑定，安全）
  • white-list=true：开启白名单（建议一直开）

3.2 修改后如何生效

  • 修改 server.properties 后需要重启：

- docker compose restart mc

注意：

  • 有些设置启动时读取，不重启不会生效。

---

4. 如何添加/删除玩家白名单、设置 OP（管理员）

4.1 “玩家名称”是什么？

  • 就是玩家在进入服务器时显示的游戏名（Game Profile Name）。
  • 当 online-mode=true 时，玩家名来自正版账号，基本不可伪造。

4.2 常用操作（推荐用 mcctl/控制台，而不是手改 json）

假设你已经配置好 RCON（并且容器里启用 RCON），常见做法：

  • 查看白名单：mcctl whitelist-list
  • 添加白名单：mcctl whitelist-add 玩家名
  • 移除白名单：mcctl whitelist-remove 玩家名
  • 给予管理员：mcctl op 玩家名
  • 取消管理员：mcctl deop 玩家名

如果你偶尔需要直接进控制台执行命令（不依赖 RCON），也可以：

  • docker attach mc-mc-1（进入控制台，退出用 Ctrl+P 再 Ctrl+Q）
  • 或者用 docker exec 执行镜像内自带的 rcon 工具（视镜像配置而定）

---

5. 连接方式：公网直连 + 域名（你现在在用的方案）

你现在的模式是：

  • 服务器开放 25565/tcp
  • 域名做 A 记录指向服务器公网 IP

    • -（可选）用 SRV 记录让玩家只填域名就能自动带端口

玩家连接（在游戏里“多人游戏”添加服务器）：

  • 如果 SRV 记录正确：直接填 你的域名
  • 如果没有 SRV 或不确定：填 你的域名:25565 或 公网IP:25565

安全建议：

  • 只开放 25565/tcp 给公网
  • 不要把 25575（RCON）开放到公网，保持仅本机/容器网络可访问

---

6. 皎月连 / NatPierce 你还需要吗？

6.1 你截图里的 natpierce 容器是什么状态

  • 它现在是 Exited (137)，说明已经不在运行。

6.2 用公网直连 vs 虚拟网段（皎月连）怎么选

  • 公网直连优点：最简单、最少环节、延迟最稳定（只要线路 OK）
  • 虚拟网段/穿透优点：不用开公网端口、某些网络环境下更容易联机

你目前已经能通过域名直连成功，所以推荐：

  • 公网直连作为主方案
  • NatPierce 作为“备用方案”，需要时再起容器

---

7. 日常运维：查看状态、日志、重启、更新

7.1 常用状态命令

在 /opt/mc：

  • docker compose ps：看容器是否 healthy
  • docker compose logs -f mc：实时看日志
  • docker stats --no-stream mc-mc-1：看 CPU/内存占用

7.2 重启/停止/启动

  • docker compose restart mc
  • docker compose stop mc
  • docker compose up -d

7.3 更新“更新的是什么”

你的更新一般包含三块：

  1. 镜像更新：itzg/minecraft-server 镜像更新（脚本/Java/辅助工具）
  1. Fabric/MC 服务端文件更新：由镜像启动流程根据你指定版本去下载/更新
  1. mods 更新：你手动替换 /opt/mc/data/mods 下的 jar

常用镜像更新命令：

  • docker compose pull
  • docker compose up -d

注意：

  • 更新镜像前先备份（尤其是 world）。

8. 备份策略（保留 7 天）应该怎么做

推荐备份包含：

  • world/（必须）
  • config/（建议）
  • server.properties、whitelist.json、ops.json

备份目录建议：

  • /opt/mc/backups/

备份动作建议（原则）：

  1. 先让服务器“落盘保存”
  1. 再打包备份
  1. 删除 7 天前的备份文件

你已经有 mcctl 脚本（或后续会完善）时：

  • 重点关注：

- 是否能触发保存（save-all）

- 备份文件命名（含日期时间）

- 清理策略（只留最近 7 天）

9. 如何添加/删除 Mod（服务端应该怎么操作）

9.1 你需要先搞清“这个 mod 属于哪类”

  • 服务端必装：功能型、内容型（生物、方块、任务、战斗系统等）
  • 客户端可选：UI、美化、性能、地图显示等（不影响服务器逻辑）

经验法则：

  • 你们整合包里的 mod，大概率是“服务端 + 客户端都要一致”。

9.2 添加 Mod（推荐流程）

  1. 停服：docker compose stop mc
  1. 把 jar 放进：/opt/mc/data/mods/

    2. 3)（如果有）同步对应配置到：/opt/mc/data/config/

  1. 启动：docker compose up -d
  1. 看日志确认没有依赖缺失/版本不兼容

9.3 删除 Mod（更谨慎）

  1. 停服
  1. 从 mods/ 移除 jar
  1. 删除该 mod 的配置（可选，先备份再删）
  1. 启动并观察日志

注意：

  • 部分 mod 会把方块/实体写进存档，删除可能导致世界报错或丢数据。
  • 重要操作前先备份 world。

10. Docker 清理：删除容器/镜像/网络/卷（含你截图里的对象）

建议：先确认哪些是“游戏服在用的”，不要误删 world。

10.1 删除某个容器（不会删镜像）

  • 删除 natpierce 容器：

- docker rm natpierce

  • 删除 mc 容器（不建议直接 rm，建议用 compose 管）：

- docker compose down（在 /opt/mc）

10.2 删除镜像

  • 删除 natpierce 镜像：

- docker rmi xiyu505/natpierce:latest

  • 删除 mc 镜像（不建议随便删，除非你确定要重拉）：

- docker rmi itzg/minecraft-server:stable-java21-jdk

10.3 删除网络

  • 删除 mc_default（需要先 down 容器）：

- docker network rm mc_default

  • 一般不用手删，docker compose down 会自动处理。

10.4 删除卷（⚠️慎重）

  • 你的截图里 natpierce_data 卷属于皎月连：

- docker volume rm natpierce_data

⚠️ 注意：

  • MC 的世界存档通常在宿主机目录 /opt/mc/data（bind mount），不是 docker volume（所以别删错）。

10.5 一键清理“无用资源”（谨慎用）

  • 清理未使用镜像/网络/构建缓存：

- docker system prune -a

⚠️ 这条会删很多东西，先确认你不需要保留旧镜像回滚。

11. 你能在游戏里改什么？哪些必须在服务器改？

11.1 游戏内（有 OP 后）能改的

  • /gamerule、/difficulty、/time、/weather、/tp、/give 等
  • 部分模组提供的管理命令（看模组说明）

11.2 必须在服务器文件/容器层面改的

  • server.properties（online-mode、端口、视距、最大玩家等）
  • mod 安装/删除（/opt/mc/data/mods）
  • 大量 mod 配置（/opt/mc/data/config，常常需要重启）
  • 版本升级（镜像、Fabric、MC 版本）

12. 常见问题快速排查（你今天就遇到过的那类）

12.1 容器 healthy 但玩家进不来

  • 检查端口：ss -lntp | grep 25565
  • 检查安全组/防火墙：确认放行 25565/tcp
  • 看日志：docker compose logs --tail=200 mc

12.2 日志里出现 ModResolutionException / Incompatible mods

  • 多半是：

- mods 版本不匹配（MC/Fabric/依赖）

- jar 文件名/编码损坏

  • 处理：

- 回滚到上一份备份

- 逐步移除最近新增的 mod 排查

12.3 RCON 连不上（mcctl 报 connect refused）

  • 确认 server.properties：enable-rcon=true、端口/密码正确
  • 确认容器确实启动并监听 25575（日志里会有 RCON running）
  • 不要把 rcon 暴露到公网

  1. 把 online-mode=true + whitelist=true 固化（减少误操作）
