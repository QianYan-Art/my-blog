背景

本次调整围绕 api.qianyan.me 的远程 LM Studio API 访问链路展开。当前链路为：

调用方 -> api.qianyan.me -> 服务器 Nginx -> frps TCP 代理端口 -> 本地 Windows 工作站 frpc -> LM Studio

主要目标：

  • 降低大提示词、图片、视频请求上传时的代理层阻塞风险。
  • 修复 Nginx 与 LM Studio 上游之间偶发慢响应问题。
  • 更新并规范服务器端 frp 更新流程。
  • 清理服务器历史安装残留。
  • 梳理 EdgeOne 与本地工作站侧的后续优化方向。

本文不记录 SSH 私钥、frp token、API key 等敏感信息。

服务器端 Nginx 调整

配置文件：

/etc/nginx/sites-enabled/lmstudio-api.conf

已确认或保留的关键配置：

client_max_body_size 200m;
client_body_timeout 1800s;
send_timeout 1800s;

proxy_connect_timeout 60s;
proxy_send_timeout 1800s;
proxy_read_timeout 1800s;

proxy_buffering off;
proxy_request_buffering off;
add_header X-Accel-Buffering no;

含义：

  • 单次 HTTP 请求体上限当前为 200MB。
  • 上传、上游发送、上游读取超时均支持 30 分钟。
  • proxy_request_buffering off 避免 Nginx 先完整缓存大提示词、图片或视频后再转发。
  • proxy_buffering off 与 X-Accel-Buffering no 有利于流式响应实时返回。

本次实际修改：

proxy_set_header Host $proxy_host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header Connection "";

调整原因：

  • 测试发现上游收到 Host: api.qianyan.me 时存在偶发 10 秒以上慢响应。
  • 将上游 Host 改为 $proxy_host 后，LM Studio/本地上游收到的是代理目标主机信息，减少公网域名 Host 带来的异常行为。
  • 添加 X-Real-IP 便于上游日志追踪。
  • 清空 Connection 头，避免不必要的 hop-by-hop 头影响代理链路。

备份位置：

/root/nginx-backups/

frp 服务端更新与脚本化

服务器端 frp 已从 0.67.0 更新到 0.68.1。

当前二进制：

/usr/local/bin/frps
/usr/local/bin/frpc

systemd 服务：

/etc/systemd/system/frps.service

服务启动方式：

/usr/local/bin/frps -c /etc/frp/frps.toml

新增服务器端更新脚本：

/usr/local/sbin/update-frp.sh

脚本能力：

  • 从 GitHub 官方 fatedier/frp release 获取最新稳定版或指定版本。
  • 根据服务器架构自动选择 Linux amd64/arm64 等包。
  • 下载后校验 release asset 的 SHA256 digest。
  • 使用新包内 frps verify -c /etc/frp/frps.toml 校验配置。
  • 备份旧二进制后替换。
  • 重启 frps 并确认服务 active。
  • 如果重启失败，尝试恢复旧二进制。

常用命令：

/usr/local/sbin/update-frp.sh
/usr/local/sbin/update-frp.sh v0.68.1
FRP_FORCE=1 /usr/local/sbin/update-frp.sh

/root/frp-backups/

服务器 IPv4 优先

服务器当前没有公网 IPv6 能力，但系统之前存在 IPv6 优先风险。已改为 IPv4 优先，不禁用 IPv6。

配置文件：

/etc/gai.conf

已启用配置：

precedence ::ffff:0:0/96  100

作用：

  • 服务器主动访问双栈域名时优先使用 IPv4。
  • 避免无可用 IPv6 路由时先尝试 IPv6 再回退 IPv4，造成额外等待。
  • 不影响未来服务器获得 IPv6 后继续使用 IPv6。

备份位置：

/root/nginx-backups/gai.conf.bak-20260423-ipv4-priority

EdgeOne 配置建议

原则：

API 域名与普通网站域名应拆分规则，不要用泛域名规则统一处理。

API 域名规则

匹配条件：

HOST 等于 api.qianyan.me

建议动作：

  • 缓存：不缓存。
  • 查询字符串：全部保留。
  • 图片处理：关闭。
  • 文件优化：关闭，尤其是 Gzip/Brotli 不建议对 SSE 流式 API 开启。
  • HTTP/2：保持开启。
  • WebSocket：普通 OpenAI 兼容接口、视觉图片/视频上传不依赖 WebSocket，可不单独开启。
  • 最大上传大小：EdgeOne 可保留 800MB，但服务器 Nginx 当前上限是 200MB。

说明：

传图和传视频通常通过 HTTP POST 请求体完成，不依赖 WebSocket。WebSocket 只在实时双向协议、Realtime API、实时语音/视频流等场景才需要。

普通网站 WebSocket 规则

匹配条件建议：

HOST 等于 huohu.qianyan.me
HTTP 请求头 Upgrade 等于 websocket

建议动作：

  • 开启 WebSocket。
  • 最大连接超时时长设置为 1800 秒，即 30 分钟。
  • 不要把 WebSocket 长连接规则套到所有 *.qianyan.me。

静态资源缓存规则

适用于普通网站，不适用于 API 域名。

示例：

  • 图片后缀 jpg/png/gif/bmp/svg/webp 浏览器缓存 1 小时可以保留。
  • 动态脚本后缀 php/jsp/asp/aspx 不缓存可以保留。
  • 这些规则应限制在普通网站 Host 下，不要影响 api.qianyan.me。

IPv6

当前源站服务器没有 IPv6。

建议：

  • 如果 EdgeOne 只是 DNS 托管，IPv6 开关基本不影响当前 API 访问。
  • 如果 EdgeOne 开启代理/加速，开启 IPv6 后客户端可通过 IPv6 访问 EdgeOne，再由 EdgeOne IPv4 回源。
  • 为稳定优先，API 域名可先关闭 EdgeOne IPv6。
  • 如果未来有 IPv6-only、移动网、校园网调用方，再单独测试开启。

Windows 工作站与 NVIDIA 建议

本地工作站为 Windows，运行 LM Studio 与 frpc。

NVIDIA 控制面板建议：

  • 为 LM Studio.exe、lms.exe 或相关 llama/server 子进程设置程序配置。
  • 电源管理模式：优先最高性能。
  • CUDA - GPU：指定 NVIDIA 显卡或全部。
  • OpenGL 渲染 GPU：指定 NVIDIA 显卡。
  • 垂直同步、最大帧率、低延迟模式、抗锯齿等对推理性能基本无关，默认即可。

Windows 电源建议：

  • 电源模式设为最佳性能。
  • PCI Express 链接状态电源管理设为关闭。
  • 工作站用途建议关闭睡眠/休眠，只关闭显示器。

HDMI 说明：

  • HDMI 线一直插着，只关闭显示器电源，通常不会明显影响 LM Studio 推理性能。
  • 4K 高刷、HDR、G-Sync 可能增加空闲功耗，但对模型推理吞吐影响很小。

真正影响 LM Studio 性能的因素：

  • 模型量化规格。
  • 上下文长度。
  • GPU offload 层数。
  • 显存是否足够。
  • CUDA/GPU 后端是否启用。
  • 是否发生内存或显存换页。

当前验证结果摘要

frp 恢复后链路采样：

frp 直连 127.0.0.1:61423/v1/models：约 0.06s ~ 0.69s
Nginx 本地 https://api.qianyan.me/v1/models：约 0.58s ~ 1.29s
HTTP 返回码：401

说明：

  • 401 表示链路可达且进入 API 鉴权层，符合预期。
  • 当前主要不是连接失败，而是偶发延迟波动。
  • 后续若继续优化上传速度，应重点观察本机代理/TUN 直连规则、frpc 传输参数、运营商链路质量和 LM Studio 上游响应。

后续待办

  • 在本地代理软件中将 api.qianyan.me 与服务器 IP 加入直连规则，避免 TUN/fake-ip 影响测速和真实调用。
  • 如确实需要上传超过 200MB 的视频，将 Nginx client_max_body_size 从 200m 提高到目标值。
  • 评估是否为 61423 增加更严格的访问限制，避免绕过 Nginx 直接访问 frp 暴露端口。
  • 如果确认不用 LXD，可进一步卸载 snap/lxd/snapd 相关残留。
  • 如果 Nginx 旧 worker 长期处于 shutting down，可在低峰期重启 Nginx 清理旧进程。
