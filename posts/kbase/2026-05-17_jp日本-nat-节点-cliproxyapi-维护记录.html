1.1 当前路由策略

/root/cliproxyapi/config.yaml 当前已启用：

routing:
  strategy: "fill-first"
  session-affinity: true
  session-affinity-ttl: "24h"

目的：

  • fill-first：优先把请求压到同一批可用账号，尽量吃满单账号上游缓存。
  • session-affinity: true：同一会话尽量回到同一账号，减少会话在多账号之间漂移导致的缓存命中下降。
  • session-affinity-ttl: "24h"：把会话到账号的绑定保留更久，适合长对话、同一项目持续编码、同一线程反复追问等场景。

副作用：

  • 账号利用率会更偏向前面的可用账号，不再像 round-robin 那样平均分散。
  • 如果最前面的账号频繁触发冷却，流量仍会自动落到后面的可用账号，不会卡死。

2. 服务职责

cliproxyapi.service：

  • 真正的 cli-proxy-api 进程。
  • 当前监听 127.0.0.1:59998。
  • 不直接暴露公网。

cliproxyapi-edge-proxy.service：

  • 前置 HTTP 代理。
  • 当前监听 0.0.0.0:59999。
  • 负责把公网请求转发给 SSE filter 127.0.0.1:59997（2026-05-17 之前直连 127.0.0.1:59998）。
  • 对长耗时图片请求做保活，对 chunked request body 做兼容。

cliproxyapi-filter.service（2026-05-17 新增）：

  • SSE 流过滤器，监听 127.0.0.1:59997，转发到 127.0.0.1:59998。
  • 实时剥离 cliproxyapi OAuth 路由返回的 "obfuscation" 字段，避免 codex 严格 JSON 解析器报 error decoding response body。
  • 纯 TCP 转发，不干预 edge-proxy 的长请求保活逻辑。服务文件：/root/cliproxyapi/filter_proxy.py。

    • /etc/systemd/system/cliproxyapi.service.d/edge-proxy.conf：

[Unit]
Wants=cliproxyapi-edge-proxy.service cliproxyapi-filter.service

用途：

  • 每次 cliproxyapi.service 被日更脚本重新 start 后，会顺带把 cliproxyapi-edge-proxy.service 一起拉起。
  • 避免 core 服务恢复后，前置代理仍然保持 dead。

3. 配置要点

/root/cliproxyapi/config.yaml 当前关键字段：

  • port: 59998
  • host: "127.0.0.1"
  • remote-management.allow-remote: true
  • remote-management.secret-key: ...
  • oauth-model-alias.vertex: ...

结论：

  • cliproxyapi 只接受本机前置代理访问。
  • 公网不要额外映射 59998。
  • 远程管理功能依赖 remote-management.secret-key，EasyCLI Remote 模式填写的是这个管理密码，不是 API key。
  • 当前已对 vertex 来源模型加 vtx/ 前缀别名，用于和其他来源区分。

4. 图片链路修复记录

4.1 2026-04-27 文生图 524

现象：

  • POST /v1/images/generations 调用 gpt-image-2 时，公网请求可能在约 15 秒处被前置 TencentEdgeOne 断开，表现为 524。

原因：

  • 实际生图耗时通常在 30 到 100 秒之间。
  • 前置 CDN 在首字节返回前就超时。

修复：

  • 增加 cliproxyapi-edge-proxy.service。
  • 对长耗时图片请求先提交 200 并持续写空白 keepalive，等待上游真正完成。

4.2 2026-04-27 客户端 prompt is required

现象：

  • Android / RikkaHub 等客户端上传请求后，返回 prompt is required。

原因：

  • 某些客户端使用 Transfer-Encoding: chunked。
  • 若前置层没正确读取 chunked body，会把请求转成空 body。

修复：

  • 在前置代理中实现 chunked request body 读取。

4.3 2026-05-11 图生图 failed to edit image: 524

现象：

  • RikkaHub 更新图生图能力后，上传图片和提示词返回 failed to edit image: 524。

原因：

  • 新版 RikkaHub 图生图走的是 POST /v1/images/edits。
  • 旧版前置代理只给 POST /v1/images/generations 做保活。

修复：

  • 前置代理现在对以下长耗时路径都做保活：

    • - POST /v1/images/generations - POST /v1/images/edits - POST /v1/images/variations

5.1 代理层补丁（2026-05-17 晚间）

这次进一步确认并修复了两类代理层问题：

  1. filter_proxy.py
  • 旧逻辑会丢掉“上游响应首包中已经带出来的 body”
  • 结果是当上游返回短错误响应（如 401 / 502）时，下游经常只收到响应头，正文不完整，表现为连接提前关闭、curl: (18) 之类的错误
  • 现已修复为：拆分上游响应头/body，并把首包 body 一并转发
  1. cliproxyapi_edge_proxy.py
  • 已补 IncompleteRead 兜底
  • 避免边缘代理在读取流式 responses 时因上游读不完整直接抛 502

说明：

  • 这两处修复主要提升错误响应与流式转发的稳定性
  • 目前 Codex + suyuan 的真实使用路径可用
  • 但对“极简手写 curl 直打 /v1/responses”这类请求，core provider 选择逻辑仍可能返回 unknown provider for model，这不影响当前 Codex 使用

6. 与每日自动更新的关系

/root/update-cliproxyapi.sh 当前已升级，除了原有逻辑，还会额外备份和恢复自定义代理层文件。

原有逻辑：

  • 备份 /root/cliproxyapi/config.yaml
  • 备份 /root/cliproxyapi/auths/
  • 更新 /root/cliproxyapi/ 目录下程序
  • 恢复 config.yaml 与 auths/

新增逻辑：

  • 备份 /root/cliproxyapi-edge-proxy/
  • 备份 /root/cliproxyapi/filter_proxy.py
  • 备份 /etc/systemd/system/cliproxyapi-edge-proxy.service
  • 备份 /etc/systemd/system/cliproxyapi-filter.service
  • 备份 /etc/systemd/system/cliproxyapi.service.d/edge-proxy.conf
  • 更新后自动恢复上述文件
  • 更新后自动 enable --now cliproxyapi-edge-proxy.service cliproxyapi-filter.service

这意味着：

  • port: 59998
  • host: "127.0.0.1"

会随着 config.yaml 恢复而保留。

因此 routing.strategy: "fill-first"、session-affinity: true、session-affinity-ttl: "24h" 也会随着 config.yaml 恢复而保留。

同理，oauth-model-alias.vertex 下的 vtx/ 前缀映射也会随着 config.yaml 恢复而保留。

前置代理相关文件位于 /root/cliproxyapi/ 目录外：

  • /root/cliproxyapi-edge-proxy/cliproxyapi_edge_proxy.py
  • /etc/systemd/system/cliproxyapi-edge-proxy.service
  • /etc/systemd/system/cliproxyapi.service.d/edge-proxy.conf

现在结论变成：

  • 安装器本身仍不会主动维护这些文件
  • 但本机 /root/update-cliproxyapi.sh 已经会把它们一起备份并恢复
  • 只要继续通过这份更新脚本做日更，前置代理与 filter 层不会像之前那样容易在更新后丢失

7. 备份与灾难恢复

如果不用 /root/update-cliproxyapi.sh，而是手动更新或整机灾难恢复，仍然需要特别关注以下文件：

  • /root/cliproxyapi-edge-proxy/cliproxyapi_edge_proxy.py
  • /etc/systemd/system/cliproxyapi-edge-proxy.service
  • /etc/systemd/system/cliproxyapi.service.d/edge-proxy.conf
  • /root/cliproxyapi/filter_proxy.py（2026-05-17 新增）
  • /etc/systemd/system/cliproxyapi-filter.service（2026-05-17 新增）

所以如果发生：

  • 整机重装
  • 手动误删目录
  • 做整机级灾难恢复

除了恢复 /root/cliproxyapi/config.yaml 与 auths/，还要单独恢复上述文件，然后执行：

systemctl daemon-reload
systemctl enable --now cliproxyapi-edge-proxy.service cliproxyapi-filter.service
systemctl restart cliproxyapi.service

9. 一句话原则

这台 JP 节点上的 cliproxyapi 现在是“core 只监听 localhost，公网只经过 edge-proxy → filter”的结构。端口 59997=filter, 59998=core, 59999=edge。以后不要再把内部端口直接暴露出去，也不要删掉 edge-proxy 和 filter 的文件而不留恢复记录。
