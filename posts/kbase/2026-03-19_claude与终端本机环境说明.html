2026-03-198 min

这份文档用于记录阿颜这台 Windows 机器上命令行相关环境的当前事实，重点覆盖：

  • claude
  • git
  • pwsh
  • node / npm

目标是让后续排障时先看清楚“命令真实入口在哪、为什么能用、为什么有时会报错”，避免重复折腾。

  • claude 使用官方独立可执行文件：

    • - C:\Users\lenovo\.local\bin\claude.exe

  • 现在的 claude 是官方独立安装路径
  • claude 在 Windows 下除了需要 claude.exe 本身，还需要可用的 Git Bash

这次问题的根因

本次“终端里找不到 Claude / Claude 启动报 Git Bash 错”实际上是两个问题叠加：

第一层：claude.exe 已换位置，但 PATH 没跟上

新入口在：

  • C:\Users\lenovo\.local\bin\claude.exe

而这个目录最开始没有进入你常用终端的 PATH，所以新开的终端里直接输入 claude 会失败。

第二层：Claude 还需要 Git Bash，不是任意 bash 都行

在 Windows 上，claude 不能只找到 git.exe，还需要可用的 Git Bash。

本机里如果直接执行：

where.exe bash

很可能先命中的是：

  • C:\Windows\System32\bash.exe
  • C:\Users\lenovo\AppData\Local\Microsoft\WindowsApps\bash.exe

这类 bash.exe 不是 Claude 需要的 Git Bash，通常对应的是 WSL 或 Windows App 映射入口。

因此即使 claude --version 正常，执行 claude 进入交互模式时仍可能报：

  • 需要 Git Bash

本次已完成的修复

1. PowerShell profile 已补充 Claude 可执行目录

文件：

  • C:\Users\lenovo\Documents\PowerShell\Microsoft.PowerShell_profile.ps1

已做的事情：

  • 把 C:\Users\lenovo\.local\bin 纳入会话级 PATH 前置
  • 保留 C:\Users\lenovo\scoop\shims 的优先级

作用：

  • 新开的 PowerShell 会更稳定地直接识别 claude

2. PowerShell profile 已补充 Claude 所需 Git Bash 路径

同一个 profile 中已加入候选路径检测逻辑，优先设置：

  • C:\Users\lenovo\scoop\apps\git\current\bin\bash.exe

作用：

  • 避免 Claude 错把 WSL 的 bash.exe 当成 Git Bash

3. 用户级环境变量已落盘

真实用户注册表 HKCU\Environment 已确认存在：

  • CLAUDE_CODE_GIT_BASH_PATH

    • - C:\Users\lenovo\scoop\apps\git\current\bin\bash.exe

  • Path

    • - %USERPROFILE%\AppData\Local\Microsoft\WindowsApps;%USERPROFILE%\.local\bin

作用：

  • 即使不是从 PowerShell profile 启动的终端
  • 即使是 cmd
  • 即使是某些 IDE 内置终端

只要是新开的进程，也能更稳定地识别：

  • claude
  • Claude 所需 Git Bash 路径

当前推荐的事实判断方式

排查 claude 问题时，不要只看单一命令，要分层判断。

第一层：Claude 命令是否存在

where.exe claude
claude --version
Get-Command claude -All | Select Source,Version

预期：

  • 命中 C:\Users\lenovo\.local\bin\claude.exe

第二层：Claude 依赖的 Git Bash 是否已绑定

echo $env:CLAUDE_CODE_GIT_BASH_PATH
Test-Path $env:CLAUDE_CODE_GIT_BASH_PATH

预期：

  • 环境变量存在
  • 指向 C:\Users\lenovo\scoop\apps\git\current\bin\bash.exe

第三层：Git 主链路是否正常

where.exe git
git --version
git lfs version

预期：

  • 命中 C:\Users\lenovo\scoop\shims\git.exe

第四层：不要用 where.exe bash 直接判断 Claude 能不能跑

因为本机默认能看到的 bash.exe 很可能先是：

  • WSL / WindowsApps 的 bash

而不是：

  • Git for Windows 的 bash

对 Claude 来说，真正关键的是：

  • CLAUDE_CODE_GIT_BASH_PATH 是否正确

后续维护准则

准则 1：不要再把 Claude 当成 npm 全局包去找

后续如果再出现：

  • npm list -g 里没有 claude
  • npm root -g 下看不到 claude

先不要误判为 Claude 被删了。

优先检查：

where.exe claude
claude --version

准则 2：Claude 升级后优先检查两个位置

优先检查：

  • C:\Users\lenovo\.local\bin\claude.exe
  • CLAUDE_CODE_GIT_BASH_PATH

如果官方后续升级导致路径变化，先修这两层，再考虑其他问题。

准则 3：Git 仍默认维护 Scoop 这套

后续如需维护 Git：

  • 默认继续维护 C:\Users\lenovo\scoop\shims\git.exe
  • 不要把 Claude 的 Git Bash 绑定回旧的 D:\jetbrains\git\Git\bin\bash.exe，除非 Scoop Git 明确失效

准则 4：新开终端再验证

环境变量写入注册表后，旧终端不会自动刷新。

正确做法：

  1. 关闭旧终端
  2. 新开 PowerShell / cmd / IDE 终端
  3. 再执行验证命令

推荐的最小自检命令集

where.exe claude
claude --version
echo $env:CLAUDE_CODE_GIT_BASH_PATH
where.exe git
git --version
git lfs version
where.exe pwsh
pwsh -v
where.exe node
node -v
npm -v
npm config get prefix

不建议的危险操作

在没有新证据前，不建议直接做这些事：

  • 直接卸载 Node，只因为 claude 不在 npm -g 里
  • 直接删除 C:\Users\lenovo\.local\bin\claude.exe
  • 直接删除旧 Git 目录，只因为 Claude 报 Git Bash 错
  • 仅凭 where.exe bash 的结果，就断定 Claude 环境一定有问题
  • 未重新打开终端就反复修改环境变量

给后续模型或维护者的简短指令

如果你是后续接手的模型或人工维护者，请先接受以下前提：

  • 不要默认假设 Claude 仍然是 npm -g 安装
  • 不要把 claude 找不到和 git.exe 找不到混为一谈
  • 在 Windows 上，Claude 需要单独确认 Git Bash
  • 优先检查：

    • - where.exe claude - claude --version - CLAUDE_CODE_GIT_BASH_PATH - where.exe git - git --version

  • 只有在这几层证据都不成立时，才考虑更大范围的重装或回滚
