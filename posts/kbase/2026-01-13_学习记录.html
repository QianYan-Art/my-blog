b-学习记录

目标：以后快速回忆“专家是什么、top-k 是什么、为什么 A3B/active 不等于显存、以及在 LM Studio 里怎么估显存”。

1. 术语速查：专家、共享权重、KV cache、临时 buffer

1.1 专家（Experts）

  • MoE（Mixture-of-Experts）模型的 MLP/FFN 部分会被拆成很多“专家”。
  • 每层都有一个专家池（expert_count / num_experts），例如 128、160、512……
  • 推理时对每个 token 进行路由，从专家池里选 top-k 个专家参与计算（expert_used_count / num_experts_per_tok）。

1.2 共享权重（Shared / non-expert weights）

不管选哪个专家，这些都会用到，常见包括：

  • Attention（Q/K/V/O 等）
  • Embedding / Unembedding（输入嵌入/输出投影）
  • Norm、Router 等小模块

直觉：**“不是专家 MLP 的那一部分”**基本都属于共享/非专家权重。

1.3 KV cache（最关键的显存大头，和上下文线性相关）

  • 为了生成下一个 token 不重算历史，注意力会把每层的 K/V 存起来。
  • KV cache 体积与上下文长度 n_ctx 线性增长。
  • KV cache 跟“模型有多少参数”不是同一回事，很多时候显存爆炸就是 KV cache 导致。

1.4 临时 buffer（workspace / 中间张量）

  • 推理引擎为了矩阵乘、注意力、softmax、prefill 等会申请临时工作区。
  • 大小跟实现、FlashAttention、batch、并发等相关，通常是 几百 MB 到 1–2GB量级波动。
  • 这块也不等于参数量/active 参数量。

2. MoE “专家数量”到底指什么？

聊天里澄清了：大家口头说“专家数量”可能指两种不同的数：

  1. 每层专家池大小（expert_count / num_experts）
  • 例：GLM-4.6 routed experts = 160（总池），shared = 1（共享专家）
  1. 每 token 激活专家数（top-k / expert_used_count / num_experts_per_tok） ✅（你最关心的“每次激活多少个”）
  • gpt-oss-120b：top-4（每 token 每层激活 4 个专家）
  • GLM-4.6：top-8（每 token 每层激活 8 个 routed 专家，另有 1 个 shared expert 常驻参与）

在 LM Studio 的参数里，如果写的是 “Experts per token / top-k”，那就是第 2 种；如果写 “Experts” 且数很大（如 512），多半是第 1 种。

3. 为什么“上下文长度”设 30000 也行？为啥常见 16384/32768？

  • 能不能用：30000 完全可以，本质就是 KV cache 预留 token 数。
  • 常用 16384/32768 主要是：
    • 某些实现会做内存块对齐/补齐（可能实际分配略大于设定值）
    • 服务端并发时可能出现“总 ctx 池被 parallel 均分”的坑（llama.cpp server 常见）

实用建议：个人单会话想省事可用 32768；严格按需求设 30000 也没问题。

4. 两种“4bit”不是一回事：MXFP4 vs GGUF Q4

4.1 gpt-oss-120b 的 MXFP4（模型自身训练后量化）

  • 重点：MoE 专家/MLP 权重占 90%+，所以把 MoE 权重用 MXFP4（约 4.25 bits/param）能大幅压体积。
  • 影响的主要是：MoE layer 里的线性投影权重（也就是专家 MLP 的大矩阵）。

4.2 你选的 GGUF q4 / q8（推理侧量化）

  • 对于“专家本来就已经是 MXFP4”的模型：GGUF 的 Q4/Q8 往往主要影响 非专家部分（embedding/attention/output 等）。
  • Unsloth 的 UD-...（Dynamic）会更“聪明”地对不同张量选不同精度，不是所有都死板 Q4。

5. KV 量化会影响什么？企业会开吗？

5.1 KV 量化影响点

  • 量化的是 KV cache 本身（不是模型权重）。
  • 好处：显著减少显存/内存，长上下文收益非常大。
  • 代价：可能轻微影响输出质量；不同模型鲁棒性不同；有些实现对 FlashAttention 等有要求。

5.2 企业部署是否会开？

  • 会（常见）：为了更长 ctx、更高吞吐、更低成本，比如 vLLM 提供量化 KV cache（如 FP8 KV cache）。
  • 也可能不开：质量极敏感/模型对 KV 量化不鲁棒时，仍用 f16/bf16。

6. “120B” 与 “active 5.1B”的构成：为什么 active ≠ 显存

6.1 gpt-oss-120b 参数构成（核心逻辑）

  • 总参数（120B 名字实际约 116.83B）
  • 主要分为：
    • MLP（≈ MoE 专家）超大一坨
    • Attention、Embed/Unembed 等共享部分较小

6.2 每 token 激活 5.13B（active）怎么来的？

近似理解：

  • Attention（全用） + Unembed（生成时要用） + MLP_total × (top-k / expert_count)
  • 因为 MoE：每层 128 专家里只选 4 个 → 4/128 = 3.125% 的专家 MLP参与计算
    所以 active 是 “计算路径上被用到的参数量”。

6.3 为什么不能用 “active=3B/5B” 直接算显存？

因为显存占用还包括：

  1. KV cache：与 n_ctx 线性相关，和 active 无关
  2. GPU 常驻权重：取决于你把哪些权重 offload 到 GPU（你把专家放 CPU，那 active 里的“专家权重”就不占显存）
  3. 临时 buffer：workspace/中间张量开销，active 不覆盖
  4. 量化元数据/对齐：GGUF Q8_0 也不是精确 1 byte/param

结论：active 参数量是“算力路径指标”，不是“显存账单”。

7. 显存估算的通用公式（最实用）

7.1 总显存 = 三块相加

显存 ≈ GPU 常驻权重 + KV cache + 临时/预留 buffer

7.2 KV cache 的可计算公式（FP16）

设：

  • L = 参与 KV 的层数（全注意力层数）
  • Hkv = num_key_value_heads
  • Dh = head_dim
  • T = n_ctx（上下文 token 数）
  • bytes = 每个 KV 元素字节数（FP16=2）

则：

  • 每 token 每层元素数 = 2 × Hkv × Dh（K 和 V 两份）
  • KV bytes = L × T × (2 × Hkv × Dh) × bytes

关键点：KV cache ∝ T（上下文线性）。

7.3 Qwen3-Next（80B-A3B）在 Q8_0 + ctx=30000 + KV不量化 的估算思路

  • Qwen3-Next 是“混合注意力”：每 4 层 1 层全注意力 → L_full = 48/4 = 12
  • Hkv=2，Dh=256
  • FP16 KV：bytes=2

每 token 每层 bytes：

  • 2 × 2 × 256 × 2 = 2048 bytes 整模型每 token：
  • 2048 × 12 = 24576 bytes ctx=30000：
  • 24576 × 30000 ≈ 0.69 GiB（约 0.7GiB）

然后再加：

  • GPU 常驻“非专家权重”体积（取决于你是否把 experts 放 CPU）
  • 临时 buffer（通常 1–2GiB 预留更稳）

这就是为什么最后会得到一个比 “A3B≈3GB” 更大的显存数：KV + buffer 不是 active 能覆盖的。

8. 实操结论（你部署时最该盯的旋钮）

  1. n_ctx：显存杀手，尤其 KV 不量化时
  2. KV cache 是否量化：长上下文强烈建议至少 q8
  3. MoE experts 放 CPU / GPU：放 CPU 省显存但可能慢；放 GPU 快但显存压力大
  4. batch/并发/FlashAttention：影响临时 buffer 和吞吐

9. 本次聊天得到的关键结论一句话版

  • “专家数量”要分清：每层专家池大小 vs 每 token 激活 top-k。
  • A3B/active 是“算力路径”，不是“显存占用”；显存主要还看 KV cache 与运行时 buffer。
  • 长上下文下，KV cache 往往是最大头；KV 量化是常见的工程化降本手段。
