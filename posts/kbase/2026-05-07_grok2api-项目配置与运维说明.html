1. 项目定位

grok2api 是一个将 Grok Web 侧能力封装为 OpenAI 兼容接口的反向代理项目。
当前用途：

  • 对外提供 /v1/models、/v1/chat/completions 等兼容接口
  • 通过导入 Grok 账号令牌，复用网页账号额度
  • 通过自定义域名对外提供统一 API 入口

本项目当前不是直接调用官方公开 API，而是走 Grok Web 侧链路，因此出口 IP、Cloudflare 风控、账号状态、Cookie / Token 有效性都会直接影响可用性。

3. 当前部署结构

当前部署由 4 部分组成：

  1. grok2api 主服务容器
    负责账号加载、OpenAI 兼容接口、请求转发。
  2. flaresolverr 容器
    负责尝试处理 grok.com 的 Cloudflare 挑战。
  3. 本机 ss-local 服务
    将一条外部 Shadowsocks 节点转为本机可用的 SOCKS5 代理。
  4. 站点反代层
    将外部域名请求转发到本机 127.0.0.1:8000。

可简化理解为：

外部请求 -> 站点反代 -> grok2api -> SOCKS5 代理 -> Grok Web

  • grok2api

  • grok-caddy

其中：

6. 关键配置说明

6.1 .env

当前 .env 主要用于以下内容：

  • 时区
  • 日志级别
  • 容器监听地址与端口
  • 数据目录、日志目录
  • 账号存储后端类型

当前账号存储后端使用本地数据库。

6.2 data/config.toml

此文件是最重要的运行配置。

当前关键字段：

  • [app]
  • [logging]
  • [features]
  • [proxy.egress]
  • [proxy.clearance]
  • [retry]
  • [account.refresh]

其中最关键的是代理配置：

当前值为：

  • proxy.egress.mode = "single_proxy"
  • proxy.egress.proxy_url = "socks5://172.18.0.1:1080"
  • proxy.egress.resource_proxy_url = "socks5://172.18.0.1:1080"
  • proxy.clearance.mode = "flaresolverr"

这表示：

  • 主业务请求走固定代理
  • 资源请求也走同一条固定代理
  • Cloudflare clearance 仍由 flaresolverr 参与处理

7. 为什么必须加代理

最初服务器直连 grok.com 时，返回结果是：

  • HTTP/2 403
  • 响应头包含 cf-mitigated: challenge

说明问题不在项目代码本身，而在服务器出口 IP 被 Cloudflare / grok.com 风控挑战。

后续验证结论：

  • 服务器原始出口：新加坡腾讯云 IP
  • 直连 grok.com：403 challenge
  • 切到日本 Shadowsocks 出口后：grok.com 返回 HTTP/2 200

因此本项目当前可用性的核心，不是“是否启动成功”，而是“代理出口是否能过 Grok 风控”。

8. 当前代理链路

8.1 Shadowsocks 来源

当前接入的是一条外部 Shadowsocks 节点。
此节点不是给全机做全局代理，而是只提供给 grok2api 使用。

8.2 本机转换方式

由于 grok2api 支持 http / https / socks5，但不直接吃 ss:// 链接，因此当前做法是：

  1. 使用 ss-local 连接外部 SS 节点
  2. 在宿主机开放一个本地 SOCKS5 入口
  3. grok2api 再连接这个 SOCKS5 地址

8.3 当前 systemd 服务

服务名：

  • shadowsocks-libev-local@grok2api

配置文件：

  • /etc/shadowsocks-libev/grok2api.json

当前监听地址：

  • 172.18.0.1:1080

这里使用 172.18.0.1 而不是 127.0.0.1，原因是：

  • grok2api 运行在 Docker 容器里
  • 容器访问不到宿主机回环地址 127.0.0.1
  • 但可以访问 Docker bridge 地址 172.18.0.1

这是本次修复里的关键点。

9. 为什么不是全局代理

当前代理只作用于 grok2api，理由如下：

  • ss-local 只是单独监听一个本地端口
  • 系统没有配置全局 http_proxy / https_proxy
  • Docker 没有统一注入全局代理环境变量
  • 只有 /opt/grok2api/data/config.toml 显式引用了 socks5://172.18.0.1:1080

结论：

  • 这条代理是项目级代理
  • 不是系统级代理
  • 不会自动影响服务器上的其他项目

12. 常见故障排查

12.1 grok.com 返回 403 challenge

现象：

  • 网页端或日志里出现 403
  • 日志出现 cf-mitigated: challenge

排查方向：

  1. 先测代理出口能否打开 https://grok.com
  2. 再看 flaresolverr 是否超时或报错
  3. 再确认账号本身是否失效

优先判断：

  • 大多数情况先怀疑出口 IP
  • 不是先怀疑账号

12.2 ERR_PROXY_CONNECTION_FAILED

现象：

  • flaresolverr 日志出现 ERR_PROXY_CONNECTION_FAILED

本项目已实际踩过的原因：

  • 容器里配置了 socks5://127.0.0.1:1080
  • 但容器无法访问宿主机回环地址

修法：

  • 改为 socks5://172.18.0.1:1080

12.3 容器正常但接口不可用

优先检查：

  1. docker compose ps
  2. docker logs grok2api
  3. curl http://127.0.0.1:8000/v1/models
  4. curl --socks5-hostname 172.18.0.1:1080 https://api.ipify.org

12.4 网页端能开，但刷新额度失败

说明：

  • 站点反代未必有问题
  • 更可能是 Grok 上游链路不通或账号状态异常

优先检查：

  • grok2api 日志里的上游状态码
  • flaresolverr 是否卡在 challenge
  • 代理出口是否变化

13. 变更 SS 节点的方法

如果后续需要更换 Shadowsocks 节点，只改两处：

13.1 修改 SS 本地服务配置

文件：

  • /etc/shadowsocks-libev/grok2api.json

改完后重启：

systemctl restart shadowsocks-libev-local@grok2api

13.2 验证端口与出口

ss -lntp | grep 1080
curl --socks5-hostname 172.18.0.1:1080 https://api.ipify.org
curl --socks5-hostname 172.18.0.1:1080 -I https://grok.com

如果 grok.com 返回 200 或正常页面响应，再继续下一步。

13.3 必要时重启项目

cd /opt/grok2api
docker compose restart grok2api

14. 不建议的做法

以下做法不建议继续使用：

  • 直接让服务器公网 IP 直连 grok.com
  • 依赖 WARP 作为主要修复手段
  • 在容器内写 127.0.0.1:1080 指向宿主机代理
  • 把这条项目代理误改成系统全局代理
  • 频繁手动改动账号池而不看日志

原因：

  • 这些做法要么不稳定，要么容易引入新的定位困难

15. 建议的维护原则

建议长期遵守：

  1. 优先保代理出口稳定，再谈账号问题
  2. 变更代理后先测 grok.com，再测项目接口
  3. 每次改配置前保留一次手工备份
  4. 只在确认需要时再动 flaresolverr
  5. 不把项目级代理扩散成全局代理
