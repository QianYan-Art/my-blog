CC-Switch 与 Codex 会话记录修复记录

记录日期：2026-05-17（更新）

本文记录本次修复 CC-Switch 切换 provider 后 Codex 聊天记录丢失问题的完整过程、改动内容与后续维护说明。

---

1. 问题描述

阿颜在 CC-Switch 中将 Codex 的 provider 从 openai 切换到 suyuan 后，发现 Codex 中看不到之前的聊天记录，只有切换后新建的会话。之后尝试手动改回 openai，Codex 报错无法启动，再改回 suyuan 也无法正常使用。

---

2. 根因分析

2.1 会话文件按 model_provider 字段过滤

Codex 的会话文件（~/.codex/sessions/ 下的 .jsonl）中，每条记录的 payload 里包含 model_provider 字段。Codex 启动时只显示与当前 config.toml 中 model_provider 值匹配的会话。

切换 provider 前的会话分布（共 284 个文件）：

model_provider文件数
openai135
(无字段)111
duckcoding24
suyuan8
custom5
88code1

切换到 suyuan 后，Codex 只显示那 8 个 model_provider: suyuan 的会话，其余全部隐藏。

2.2 config.toml 中有残留的错误配置段

手动改配置时在文件末尾残留了一段不完整的 [model_providers.openai]（只有 base_url，缺少其他必要字段），导致 Codex 启动报错。

2.3 CC-Switch 数据库中的 config 也有同样问题

CC-Switch 数据库中 suyuan provider 存储的 config 字符串末尾同样带着那个不完整的 [model_providers.openai] 段。每次通过 CC-Switch 切换 provider 时，它会把这个带问题的 config 写入 config.toml，导致问题反复出现。

---

3. 修复内容

3.1 清理 config.toml

删除了 ~/.codex/config.toml 末尾残留的 [model_providers.openai] 段。

3.2 清理 CC-Switch 数据库中 suyuan 的 config

从 suyuan provider 的 settings_config.config 字符串中移除了末尾的 [model_providers.openai] 段。

3.3 批量修改会话文件的 model_provider

将所有 284 个会话文件中的 model_provider 字段改为 suyuan：

  • 165 个文件：原有 model_provider 值替换为 suyuan
  • 108 个文件：原本无 model_provider 字段，新增了该字段
  • 11 个文件：已是 suyuan 或老格式无法安全修改，跳过

3.4 更新 CC-Switch 中官方与 suyuan 两套可切换配置

本次最终采用“双配置并存”的方式，不再把所有 Codex provider 统一写成 model_provider = "suyuan"。

当前策略：

  • OpenAI Official 保持官方配置，用于日常默认使用。
  • suyuan 单独维护为可用的第三方配置，用于按需切换。
  • duckcoding / 88code / YesCode 不作为本次主维护目标，不再强制改成 suyuan 模式。

这样可以同时满足两点：

  • 默认继续用官方，不破坏当前稳定状态。
  • 通过 CC-Switch 可以快速切到 suyuan，并且这份配置已经过实测验证。

3.5 归档会话移回活跃目录

将 ~/.codex/archived_sessions/ 下的 66 个会话文件移回 ~/.codex/sessions/ 对应的日期目录（YYYY/MM/DD/）。

3.6 补全 session_index.jsonl

session_index.jsonl 原本只有 85 条索引，补全了缺失的 199 条，最终 287 条。

3.7 Codex 认证修复（experimental_bearer_token）

requires_openai_auth = true 会劫持路由到 api.openai.com，无论 base_url 设什么。api_key 对当前 Codex 自定义 provider 也不稳定。最终确认可用的方案是：

  • suyuan 使用 experimental_bearer_token
  • base_url = "https://suyuan.4071253.xyz/v1"
  • wire_api = "responses"
  • supports_websockets = false

这套配置已经通过 codex exec ... "只输出pong" 实测返回 pong。

3.8 VPS SSE obfuscation 过滤（2026-05-17）

cliproxyapi 的 OAuth 路由在 SSE delta 事件中插入 "obfuscation":"xxx" 非标准字段，导致 codex 严格解析器报 error decoding response body。在 VPS 上部署了 filter_proxy（127.0.0.1:59997 → 127.0.0.1:59998），实时剥离该字段。edge-proxy 上游从 59998 改为 59997，完整链路不变。

详细记录见：D:\Answer\QianYan-KBase\my_server\b-节点\JP日本 NAT 节点 CLIProxyAPI 维护记录.md

5. 后续维护说明

5.1 通过 CC-Switch 切换 provider

当前已确保：

  • OpenAI Official 维持官方可用配置
  • suyuan 维持已验证可用的第三方配置

正常情况下通过 CC-Switch 在这两者之间切换，不会破坏当前默认状态，也不会让 suyuan 再次失效。

如果 CC-Switch 更新后覆盖了数据库（应用更新、重置等），需要重新检查：

  1. 打开 CC-Switch → Codex
  2. 检查 OpenAI Official 仍是官方模板
  3. 检查 suyuan 仍包含 [model_providers.suyuan] 与 experimental_bearer_token

5.2 当前推荐用法

  • 日常保持 OpenAI Official 作为默认 provider
  • 需要走 VPS 节点时，在 CC-Switch 中切到 suyuan
  • 切换后若桌面端未立即刷新，重开一次 CC-Switch 或 Codex Desktop 更稳

5.3 手动修复方法

如果以后又出现“切到 suyuan 后看不到旧会话”或 “suyuan 无法请求”，优先检查两类问题：

  1. suyuan 模板是否还是 experimental_bearer_token 方案
  2. 会话文件首行 payload.model_provider 是否仍为 suyuan

如果再次需要批量修复会话文件，可用以下脚本：

import sqlite3, json, re, os

# 修改 CC-Switch 数据库
db = r'C:\Users\lenovo\.cc-switch\cc-switch.db'
conn = sqlite3.connect(db)
cur = conn.cursor()
cur.execute("SELECT id, name, settings_config FROM providers WHERE app_type='codex'")
for row in cur.fetchall():
    pid, name, raw = row
    data = json.loads(raw)
    config = data['config']
    if 'model_provider' in config:
        config = re.sub(r'model_provider\s*=\s*"[^"]*"', 'model_provider = "suyuan"', config)
    else:
        config = config.replace('model = "gpt-5.4"', 'model_provider = "suyuan"\nmodel = "gpt-5.4"', 1)
    data['config'] = config
    cur.execute("UPDATE providers SET settings_config=? WHERE id=? AND app_type='codex'",
                (json.dumps(data), pid))
conn.commit()
conn.close()
// 批量修改会话文件（Node.js）
const fs = require('fs');
const path = require('path');

function walk(dir) {
  const results = [];
  for (const e of fs.readdirSync(dir, { withFileTypes: true })) {
    const full = path.join(dir, e.name);
    if (e.isDirectory()) results.push(...walk(full));
    else if (e.name.endsWith('.jsonl')) results.push(full);
  }
  return results;
}

for (const f of walk('C:/Users/lenovo/.codex/sessions')) {
  let content = fs.readFileSync(f, 'utf-8');
  if (content.includes('"model_provider"')) {
    content = content.replace(/"model_provider"\s*:\s*"[^"]*"/g, '"model_provider":"suyuan"');
  } else {
    const lines = content.split('\n');
    try {
      const rec = JSON.parse(lines[0]);
      if (rec.payload) {
        rec.payload.model_provider = 'suyuan';
        lines[0] = JSON.stringify(rec);
      }
    } catch {}
    content = lines.join('\n');
  }
  fs.writeFileSync(f, content, 'utf-8');
}
