两台VPS Reality SNI切换与简易HTTPS站部署记录

2026-05-0811 min

目标

为两台 VPS 上的 sing-box VLESS + Reality 节点补一个可控的最简 HTTPS 握手站，用于把 SNI/serverName 从第三方大厂域名切换为自有域名，并同时满足以下要求：

  • 不抢占现有 443
  • 尽量少占内存
  • 证书可自动续期
  • VPS 重启后可自动恢复
  • 不影响原有 komari-agent
  • 客户端尽量少改配置

---

最终设计总览

设计原则

现有节点已经由 sing-box 监听 443，因此不直接把 HTTPS 站部署到 443，而是采用下面的分工：

  • 443：继续由 sing-box 提供 VLESS + Reality
  • 80：用于 HTTP 验证、ACME、续期
  • 8443：最简 HTTPS 站，作为 Reality 的握手目标

改造后客户端需要改什么

客户端通常只需要修改节点中的 SNI/serverName：

  • 机器一：sni.4071253.xyz
  • 机器二：sni.starryocean.qzz.io

通常不需要改：

  • address
  • port
  • uuid
  • public key
  • flow

为什么 address 不一定要改成域名

address 只是客户端真正连接 VPS 的入口。

如果当前 VPS 的 IP 稳定，继续使用 IP 作为连接入口没有问题。这次改造的核心目标是把 Reality 握手所用的 SNI 从第三方域名切换为自有域名，而不是强制替换客户端连接入口。

---

机器一

基本信息

  • IP：82.40.32.123
  • SSH 端口：23333
  • 新 SNI 域名：sni.4071253.xyz
  • 原节点类型：VLESS-REALITY-443
  • 原 server_name：aws.amazon.com

实际变更

  1. 保持 sing-box 继续监听 443
  2. 安装 Caddy
  3. 新建最简静态站目录：

    4. - /var/www/sni.4071253.xyz/index.html

  4. 新建 Caddy 配置：

    5. - /etc/caddy/Caddyfile

  5. 站点端口规划：

    6. - 80：仅用于 HTTP/ACME - 8443：最简 HTTPS 站

  6. 修改 Reality 配置文件：

    7. - /etc/sing-box/conf/VLESS-REALITY-443.json

  7. 保留原有节点身份信息不变，仅切换：

    8. - tls.server_name - tls.reality.handshake.server - tls.reality.handshake.server_port = 8443

证书方案

机器一使用 Caddy 自动签发并自动续期证书：

  • 域名：sni.4071253.xyz
  • 签发机构：Let's Encrypt
  • 证书由 Caddy 自动管理

当前核心状态

  • sing-box 监听 443
  • Caddy 监听 80 和 8443
  • HTTPS 站访问地址：

    • - https://sni.4071253.xyz:8443

  • Reality 配置结果：

    • - server_name = sni.4071253.xyz - handshake.server = sni.4071253.xyz - handshake.server_port = 8443

维护要点

  • 域名必须保持 仅 DNS
  • 不要把该域名切到 CDN、EdgeOne 代理或其他反向代理模式
  • 不要关闭 80
  • 不要删除 Caddy 的证书存储目录
  • VPS 重启后无需额外人工操作，caddy 和 sing-box 可自动拉起

机器二

基本信息

  • IP：69.63.211.129
  • SSH 端口：23333
  • 新 SNI 域名：sni.starryocean.qzz.io
  • 原节点类型：VLESS-REALITY-443
  • 原 server_name：www.amazon.com

实际变更

  1. 保持 sing-box 继续监听 443
  2. 安装 Caddy
  3. 新建最简静态站目录：

    4. - /var/www/sni.starryocean.qzz.io/index.html

  4. 新建 Caddy 配置：

    5. - /etc/caddy/Caddyfile

  5. 站点端口规划：

    6. - 80：用于 HTTP/ACME - 8443：用于 HTTPS 握手站

  6. 修改 Reality 配置文件：

    7. - /etc/sing-box/conf/VLESS-REALITY-443.json

  7. 保留原有节点身份信息不变，仅切换：

    8. - tls.server_name - tls.reality.handshake.server - tls.reality.handshake.server_port = 8443

证书方案

机器二没有继续使用 Caddy 自动签发，而是改为 acme.sh + webroot 方案。

原因：

  • Debian 自带的 Caddy 2.6.2 在该机针对该域名自动签发时出现内部 panic
  • 为了降低不确定性，改用更稳的 acme.sh 管理证书签发与续期

最终方案：

  • 使用 acme.sh 为 sni.starryocean.qzz.io 申请 Let's Encrypt 证书
  • webroot 使用：

    • - /var/www/sni.starryocean.qzz.io

  • 证书部署到：

    • - /etc/ssl/starryocean/sni.starryocean.qzz.io.crt - /etc/ssl/starryocean/sni.starryocean.qzz.io.key

  • Caddy 读取上述证书文件提供 HTTPS
  • acme.sh 续期任务保存在 root 的 crontab
  • 续期后执行：

    • - systemctl restart caddy

当前核心状态

  • sing-box 监听 443
  • Caddy 监听 80 和 8443
  • HTTPS 站访问地址：

    • - https://sni.starryocean.qzz.io:8443

  • Reality 配置结果：

    • - server_name = sni.starryocean.qzz.io - handshake.server = sni.starryocean.qzz.io - handshake.server_port = 8443

维护要点

  • 域名必须保持 仅 DNS
  • 不要切到 CDN 或代理模式
  • 80 必须保持可访问，用于 acme.sh 续期
  • 不要删除：

    • - /root/.acme.sh - /etc/ssl/starryocean/

  • VPS 重启后无需手动干预，caddy 和 sing-box 可自动恢复

与需求的对应关系

本方案满足你的实际需求：

  1. VLESS + Reality 仍由 sing-box 在 443 提供服务
  2. Reality 握手使用自有域名，而不是大厂域名
  3. 握手落点是本机 8443 的真实 HTTPS 站
  4. 客户端改动极小，只需改 SNI/serverName
  5. 不破坏现有 komari-agent
  6. 支持证书自动续期与重启自动恢复

清理记录

为避免额外占用内存或磁盘，只做了低风险清理：

  • 清理 apt 缓存
  • 删除临时脚本和安装中间文件

没有做的操作：

  • 没有强行 drop_caches
  • 没有停用 komari-agent
  • 没有改动 kejilion.sh 既有逻辑
  • 没有删除系统必要服务

原因：

Linux 页缓存本身可回收，强制清缓存对长期稳定运行帮助很小，反而可能引入额外抖动。

建议保留的自检命令

下面这些命令适合作为手工核验命令使用，但只建议临时执行，不建议做成后台常驻脚本。

机器一

systemctl is-active caddy
systemctl is-active sing-box
ss -lntp | grep -E ':80 |:443 |:8443 '
curl -I https://sni.4071253.xyz:8443
echo | openssl s_client -connect 127.0.0.1:443 -servername sni.4071253.xyz 2>/dev/null | openssl x509 -noout -subject -issuer -dates

机器二

systemctl is-active caddy
systemctl is-active sing-box
ss -lntp | grep -E ':80 |:443 |:8443 '
curl -I https://sni.starryocean.qzz.io:8443
echo | openssl s_client -connect 127.0.0.1:443 -servername sni.starryocean.qzz.io 2>/dev/null | openssl x509 -noout -subject -issuer -dates
crontab -l

关于 127.0.0.1:8443 的说明

不要把下面这种命令的报错误判为站点故障：

curl https://127.0.0.1:8443

原因是：

  • 该 HTTPS 站是按域名签发证书的
  • 直接访问 127.0.0.1 时，请求里可能没有带正确的 SNI
  • 因此可能出现 TLS 握手错误

更可靠的检查方式是使用真实域名访问，或者显式带 SNI 的 openssl s_client 检查。

---

风险提醒

  1. 用于 Reality 握手的域名必须保持 仅 DNS
  2. 不要把这两个 SNI 域名切回 CDN、代理或其他中间层
  3. 不要关闭 80
  4. 不要让其他服务抢占 8443
  5. 机器二的证书续期依赖 acme.sh，不要删除 /root/.acme.sh
  6. 不要随意删除机器二的 /etc/ssl/starryocean/
  7. 不要无确认地改动 sing-box 的 server_name、handshake.server、handshake.server_port

2026-05-08 故障排查补充

现象

探针显示两台 VPS 的单核 CPU 长时间打满。

根因

根因不是 sing-box 主进程异常，也不是 caddy 或 komari-agent 异常，而是自检类探测命令被长期挂在后台运行。

#### 机器一发现的残留进程

  • /bin/sh /root/update-reality.sh
  • /bin/bash /etc/sing-box/sh/sing-box.sh info VLESS-REALITY-443
  • zsh -c /etc/sing-box/sh/sing-box.sh info VLESS-REALITY-443 | sed -n '1,20p'

#### 机器二发现的残留进程

  • bash -c echo | openssl s_client -connect 127.0.0.1:443 -servername sni.starryocean.qzz.io ...
  • /bin/bash /etc/sing-box/sh/sing-box.sh info VLESS-REALITY-443.json

这些进程会持续执行：

  • sing-box.sh info
  • openssl s_client
  • 配置读取与证书探测

最终导致单核 CPU 被长期占用。

处理动作

已手动终止上述残留进程。

处理后复查结果：

  • 高占用探测进程消失
  • 即时 CPU 采样恢复正常
  • sing-box 仍为 active
  • caddy 仍为 active
  • komari-agent 未受影响

经验结论

后续不要把自检命令用以下方式长期挂起：

  • watch
  • while true
  • nohup
  • 后台 bash -c / zsh -c
  • 自定义长期循环脚本

正确做法是：

  • 需要时手工执行一次
  • 确认结果后退出
  • 不要把它们变成长期探针

如果后续再次出现单核 CPU 异常打满，优先检查是否存在以下残留命令：

  • sing-box.sh info
  • openssl s_client -connect 127.0.0.1:443 ...
  • 自定义 update-reality.sh
